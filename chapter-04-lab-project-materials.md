# 第 4 章 实验与项目材料

## 对应材料

- 对应书稿：[04-participating-in-and-organizing-open-source-projects.md](04-participating-in-and-organizing-open-source-projects.md)
- 配套学习指南：[chapter-04-study-guide.md](chapter-04-study-guide.md)
- 配套教学指南：[chapter-04-instructor-guide.md](chapter-04-instructor-guide.md)
- 对应实验阶段：实验 4 GitHub 协作流与最小 CI
- 团队项目阶段：变更控制与最小门禁建立

## 本章任务目标

- 把团队项目真正放进“变更控制主链”，而不是继续直接修改主分支。
- 让每一组修改都经过工作项、分支、PR、评审和最低自动化检查。
- 为后续章节的项目阅读、外部贡献和项目迭代建立可信的工程底座。

## 使用建议

- 优先围绕一个边界清楚的小任务组织第一次完整流程，不要一开始就把过大改动塞进单条 PR。
- 所有流程对象都应回到真实团队仓库中，不建议只在本地做“未公开的演示流程”。
- CI 可以从最小检查开始，但必须真正接入日常变更，而不是演示当天临时补上。
- 发布与版本边界哪怕只做最小版本，也应在清单中明确写出“何时算可合并、何时算可发布”。

## 必做基础项

- 围绕至少一个明确的 `Issue` 启动一次开发，禁止直接在默认分支上完成全部修改。
- 所有成员通过 `Branch` 和 `Pull Request` 提交变更，至少完成一轮 review 和修改。
- 在仓库中建立最小自动化检查，可在 build、lint、test 中任选其一，但必须接入日常协作流程。
- 团队写出《最小工作流与评审/发布清单》，建议放入 `docs/workflow-and-release-checklist.md`。
- 在文档中明确工作项记录、分支命名、提交粒度、PR 描述、评审要求和发布前最小检查。

## 选做提升项

- 增加 `CODEOWNERS`、保护分支规则或规则集。
- 增加依赖变化检查、敏感信息检查或更完整的 CI 工作流。
- 增加 `CHANGELOG.md` 或一次可供演示的预发布记录。
- 尝试把版本号、发布说明和演示对象纳入同一轮协作闭环。

## 本章交付物

- 一份《最小工作流与评审/发布清单》，建议仓库路径为 `docs/workflow-and-release-checklist.md`。
- 至少一组 `Issue -> Branch -> PR -> Review` 记录。
- 至少一个最小 CI 工作流文件，如 `.github/workflows/ci.yml`。

## 提交证据

- `docs/workflow-and-release-checklist.md` 链接。
- 至少一个 `Issue` 链接、一个 PR 链接和对应 review 记录。
- CI 运行结果链接或截图。

## 评分尺

总分按 `100` 分计算，固定分为 `5` 个维度，每项 `20` 分。

| 评分维度 | 权重 | 不达标（`0-9`） | 达标（`10-15`） | 优秀（`16-20`） |
| --- | --- | --- | --- | --- |
| 变更控制链完整度 | `20` | 仍直接在默认分支修改，或链条明显断裂 | 已形成 `Issue -> Branch -> PR -> Review` 的最小闭环 | 主链完整，且每个对象的作用清楚可见 |
| PR 与评审质量 | `20` | PR 只堆代码，没有说明与评审依据 | PR 有基本说明，能支撑一轮评审 | 说明、边界和 review 往返都清楚，修改真正成为公共判断对象 |
| CI / Checks 接入质量 | `20` | 没有检查，或检查未进入日常流程 | 已有最小自动化检查，并在协作中使用 | 检查与主线保护明确关联，团队知道哪些检查承担门禁作用 |
| 发布与边界意识 | `20` | 只会合并，不区分发布准备 | 已区分“可合并”和“准备发布” | 发布清单、版本边界和对外说明前后连贯 |
| 团队流程可执行性 | `20` | 流程写在文档里，但实际协作仍靠临时默契 | 已能支撑团队当前迭代 | 工作项、分支、PR、评审和检查已经进入团队真实协作习惯 |

## 真实流程记录与最小模拟样例的等价判定

- 优先提交真实仓库中的流程记录，因为这最能暴露工作流是否真的进入协作。
- 如果课程阶段限制导致流程未完全走完，可用基于真实仓库对象的最小模拟记录补齐，但必须说明哪些部分是真实、哪些是补充。
- 是否使用更复杂的 CI，不作为单独加分项；关键看流程是否闭环、门禁是否真实存在。
- 只做页面截图演示而没有真实对象链，不应评为高分。

## 总体判分建议

- `90-100`：优秀。变更控制链完整，PR 与评审有质量，CI 与发布边界已经进入真实协作。
- `75-89`：达标偏强。主链成立，检查和评审基本到位，只有少量对象仍较弱。
- `60-74`：基本达标。最小流程已建立，但 PR 质量、检查接入或发布边界仍不够稳。
- `60` 以下：不达标。仍停留在“先改完再说”或“流程只在纸面上存在”的状态。
- 若缺少最小工作流说明或真实流程证据，通常不判为“完整达标”。

## 完成度示意

- 最低通过示意：能提交一条真实的最小流程链，并给出《最小工作流与评审/发布清单》初稿。
- 优秀完成示意：不仅对象齐全，而且团队已经开始把 PR、review、checks 和发布边界当作日常协作方式。

## 常见扣分点

- 仍直接修改默认分支，再把结果补成 PR。
- PR 范围过大，无法有效评审。
- CI 只在展示前运行一次，没有接入日常流程。
- 把发布理解成打标签或导出压缩包，没有对外承诺说明。

## 检查点

- 是否真正避免了“先改完再补说明”的做法。
- PR 是否包含足够信息，让修改成为可判断对象，而不只是代码堆叠。
- Review 与 Checks 是否都进入了流程，而不是只保留其一。
- 发布清单是否区分“可以合并”和“准备发布”。

## 常见问题

- 仍然直接推送默认分支，只把 PR 当成展示窗口。
- PR 过大，导致评审只能形式化通过。
- CI 只在演示前临时补一次，没有进入日常协作。
- 把发布理解成“打标签”或“提交压缩包”，没有对外说明边界。
