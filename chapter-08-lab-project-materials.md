- 团队项目阶段：从“课程内协作对象”转向“可公开参与项目”的最终整备

## 本章任务目标

- 把团队项目从“已经有代码”推进到“外部人可以理解、使用、参与和继续维护”的公开对象。
- 让学生把 README、LICENSE、贡献入口、版本、`CHANGELOG`、维护计划与 `SECURITY.md` 组织成一套相互一致的公共界面。
- 让学生完成一次最小正式发布准备，而不是只停留在“准备以后再发布”。
- 让学生理解：项目发布不是课程收尾动作，而是维护责任真正开始的起点。

## 使用建议

- 如果项目已经具备基本功能，优先围绕“对外解释与发布边界”做收束，不要在本阶段继续追加大功能。
- 如果项目尚不适合立刻公开发布，可采用“发布候选包”或“模拟发布包”方式完成本章任务，但必须保持真实仓库上下文。
- 建议先完成《项目发布与维护基线说明》初稿，再逐项补齐 README、`CHANGELOG`、维护计划与安全说明，不要边想边堆文件。
- 如果团队已经在第 7 章建立了 AI 使用规则，本章应把这些规则继续收束到发布前检查与维护计划中，而不是另起一套平行说明。

## 必做基础项

- 团队完成一次公共界面审查，至少覆盖 README、LICENSE、贡献入口、版本记录、维护计划与安全说明六类对象。
- 写出一份《项目发布与维护基线说明》，建议命名为 `project-release-and-maintenance-baseline.md`，至少说明：项目定位、当前版本状态、最小使用路径、贡献入口、支持版本、维护节奏与安全联系路径。
- 对 README 做一次面向陌生人的重写或收紧，至少回答：项目是什么、当前状态如何、最短上手路径是什么、如果继续参与应去哪里。
- 明确一次版本边界，给出版本号，并写出至少 `1` 条最小 `CHANGELOG` 条目和 `1` 份最小 release note。
- 写出一份最小维护计划，至少说明：谁在维护、多久查看问题、哪些版本仍被支持、哪些请求当前不在范围内。
- 写出一份最小 `SECURITY.md` 或等价安全说明，至少说明：漏洞报告路径、支持版本、公开披露前的基本处理方式。
- 团队完成一次发布前检查，形成一份《发布前检查记录》，说明哪些项已达成、哪些仍是已知限制。

## 选做提升项

- 在仓库中实际创建一次 draft release、预发布版本（pre-release）或课程内正式版本标签。
- 补一份《范围外请求处理说明》，整理团队如何礼貌但清楚地回应当前不接收的需求。
- 为团队项目补一份支持矩阵，例如“哪些版本仍维护、哪些版本只读归档、哪些版本已不再支持”。
- 把第 7 章形成的 AI 使用规则继续写进发布前检查或维护计划，说明 AI 相关改动在发布前需要额外确认哪些事项。
- 追加一份《项目复盘》，说明本次发布中最难回答的外部问题是什么，以及项目还缺什么。

## 本章交付物

- 一份《项目发布与维护基线说明》。
- README、LICENSE、贡献入口的整备结果。
- 一份最小 `CHANGELOG` 条目与一份最小 release note。
- 一份最小维护计划。
- 一份最小 `SECURITY.md` 或等价说明。
- 一份《发布前检查记录》。
- 一次真实 release / draft release / 版本标签，或一套完整的模拟发布包。

## 提交证据

- 仓库链接、对应分支或版本标签链接。
- README、`CHANGELOG`、维护计划、`SECURITY.md` 的文件路径。
- release note 或 release 页面截图 / 链接。
- 《项目发布与维护基线说明》与《发布前检查记录》。
- 若未实际公开发布，需附一段解释：为什么当前采用模拟发布包，以及哪些条件满足后可以转为真实发布。

## 评分尺（实验 8）

总分按 `100` 分计算，固定分为 `5` 个维度，每项 `20` 分。

| 评分维度 | 权重 | 不达标（`0-9`） | 达标（`10-15`） | 优秀（`16-20`） |
| --- | --- | --- | --- | --- |
| 公共界面完整度 | `20` | 仍停留在“代码可见”，外部人无法判断项目定位与进入路径 | README、LICENSE、贡献入口等核心对象已齐备，能回答基本问题 | 公共界面对象之间相互一致，外部人可快速形成定位、状态与进入判断 |
| 版本与发布承诺 | `20` | 只有仓库状态，没有清楚版本边界、`CHANGELOG` 或发布说明 | 已有版本号、最小变更记录与发布说明 | 版本号、`CHANGELOG`、release note 与实际交付对象清楚对应 |
| 维护计划与范围控制 | `20` | 没有维护责任、节奏或范围边界说明 | 已形成最小维护计划，能说明谁维护、多久看问题、哪些版本仍支持 | 维护节奏、范围控制与版本支持线清楚，外部人能判断项目连续性 |
| 安全与供应链基线 | `20` | 没有漏洞报告路径，不清楚支持版本，也看不出最小安全意识 | 已形成最小 `SECURITY.md` 或等价说明，并能解释基本安全边界 | 安全说明、支持版本、发布前检查与最小权限/依赖意识能形成闭环 |
| 最终公共说明质量 | `20` | 文件虽存在，但像内部记录，外部人仍难理解 | 已形成《项目发布与维护基线说明》与发布前检查记录 | 说明结构清晰、语言面向外部人、各对象能共同支撑一次正式发布判断 |

## 真实发布与模拟发布的等价判定

- 如果项目已具备公开发布条件，真实 tag、draft release、pre-release 或正式 release 都可作为有效交付方式。
- 如果项目尚未满足公开条件，也可以围绕真实仓库完成“模拟发布包”，但必须保留真实版本边界、真实 README、真实 `CHANGELOG`、真实维护计划与安全说明。
- 是否被外部人立刻采用，不作为本章直接评分项；重点是项目是否已经被组织成“可以被外部人理解和进入”的对象。
- 纯粹做视觉修饰、徽章堆砌或静态截图，而没有形成真正的公共界面和发布边界，不应评为高分。

## 总体判分建议

- `90-100`：优秀。项目已具备清楚公共界面、版本边界、维护计划和最小安全基线，可直接作为一个小型开源项目对外展示。
- `75-89`：达标偏强。主要对象齐全，发布逻辑成立，只有少量边界或说明仍显粗糙。
- `60-74`：基本达标。已形成最小发布包，但对象之间尚未完全一致，或维护 / 安全部分仍较弱。
- `60` 以下：不达标。仍停留在“仓库里有代码”，无法证明项目为什么已准备好进入公共世界。

## 完成度示意

- 最低通过示意：能交出《项目发布与维护基线说明》、最小版本说明、最小维护计划和最小安全说明，并完成一次发布前检查。
- 优秀完成示意：不仅形成完整发布包，还能让一个陌生人仅凭仓库公共界面就理解项目定位、版本边界、参与方式与持续维护状态。

## 常见扣分点

- README 只有功能介绍，没有项目当前状态和最短上手路径。
- 有版本号但没有 `CHANGELOG` 或 release note，外部人无法判断这次到底发布了什么。
- 维护计划只写“欢迎贡献”，却没有谁维护、多久响应、哪些版本还支持。
- 安全说明停留在口号，没有漏洞报告路径或支持版本边界。
- 团队把“以后再完善”写成默认答案，导致本章没有形成真正的公开发布对象。

## 检查点

- 陌生人第一次进入仓库，能否在较短时间内回答“这是什么、怎么开始、出了问题找谁、这版稳不稳定”。
- 版本号、`CHANGELOG`、release note 和实际仓库状态是否一致。
- 维护计划是否真的回答了近距离问题，而不是只写长期愿景。
- 安全说明是否最小但真实，而不是纯装饰文件。
- 整套对象是否已经从“内部团队默契”转成“外部世界可以理解的公共结构”。

## 常见问题

- 把本章任务理解成“发一个版本号就结束”，忽略公共界面整备。
- 把发布说明写成提交历史复制，而不是写给外部人看的变更说明。
- 继续把大量开发中细节留在团队聊天记录里，没有写回仓库。
- 觉得小项目不需要维护计划或安全说明。

## 与课程收束的衔接

- 本章是全书的最后一个章节级任务，目标不是继续扩展开发，而是完成“从项目开发到项目公开”的最后转换。
- 如果第 7 章已经形成 AI 规则、任务边界和人工验收要求，本章应把这些对象继续纳入发布前检查和长期维护计划。
- 本章完成后，团队项目至少应具备继续被下一届学生、外部贡献者或未来维护者接手的最小条件。
