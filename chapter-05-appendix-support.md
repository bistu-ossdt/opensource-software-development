# 第 5 章 附录支撑

## 使用说明

- 本附录服务于第 5 章的项目阅读、整体判断与行动选择任务。
- 目标是帮助读者把“浏览仓库”转化为“形成可解释判断”，而不是替代正文中的分析方法。
- 可直接复用模板，但必须结合目标项目的真实公开对象填写。
- 建议先独立完成自己的《陌生项目阅读记录》和《项目阅读摘要》初稿，再对照后文示例包，不要先照着样例反推答案。

## 第一轮阅读顺序检查单

- 是否先确认项目解决什么问题、面向谁、当前主要使用场景是什么。
- 是否看过 `README`、主页、许可证、贡献入口、最近活动与版本记录。
- 是否至少检查了一个真实 `Issue` 或 Pull Request，而不只停留在首页文字。
- 是否看过工作流、检查状态或其他工程门禁痕迹。
- 是否已经能够说出“这个项目下一步值得继续做什么”。

## 四类对象观察清单

- 入口对象：`README`、主页、安装说明、文档入口、贡献入口。
- 治理对象：`CONTRIBUTING.md`、`CODE_OF_CONDUCT.md`、`SECURITY.md`、`MAINTAINERS`、模板文件。
- 工程对象：`Issue`、Pull Request、工作流文件、检查状态、版本记录、`CHANGELOG`。
- 风险对象：依赖变化、安全说明、维护响应、版本停滞、对象之间的不一致。

## 行动选择快速判断表

| 行动选择 | 常见适用情形 | 下一步动作 |
| --- | --- | --- |
| 继续学习 | 项目值得深入，但当前上下文还不够 | 继续补项目背景、结构与实现层认知 |
| 尝试贡献 | 入口清楚、维护节奏稳定、存在真实可进入对象 | 先记录候选对象与进入前提，再在第 6 章组织第一次贡献 |
| 复用 | 项目有现实价值，但当前更关注版本与制度边界 | 继续核对版本、依赖与许可证条件 |
| 先观察 | 项目有吸引力，但维护节奏或证据链仍不稳定 | 暂不投入更多实现级精力，持续跟踪 |
| 暂缓 | 风险高于当前收益，继续投入不划算 | 停止深入，转向其他候选对象 |

## 好判断与坏判断的最小对照

```text
不够好的项目阅读摘要
- 这是一个很热门的项目，功能很多，应该值得继续参与。

更稳妥的项目阅读摘要
- 这个项目定位清楚，README 和贡献入口一致，最近版本仍在更新。
- 但第一次进入更适合先从文档或小修复观察，因为当前主线仍在快速变化。
- 因此当前行动选择是“继续学习 + 记录候选进入对象”，暂不贸然进入复杂修改。
```

## 示例包使用顺序

- 教师课堂校准时，建议按“常见问题样例 -> 达标样例 -> 高质量样例”的顺序展示。
- 学生自查时，建议先完成自己的《陌生项目阅读记录》《项目阅读摘要》和《公共入口缺口清单》，再来对照样例。
- 以下三组样例都围绕同一类轻量对象展开：一个入口清楚、活动记录可见的小型公开项目。

## 示例包 A：高质量样例

以下样例围绕同一个轻量公开项目展开，只用于校准判断质量，不要求与某个特定仓库逐字对应。

### 1. 《项目阅读摘要》样例

```md
# 项目阅读摘要

- 项目解决什么问题：为常见命令提供统一的简明示例，主要面向命令行初学者和需要快速查阅的人。
- 制度边界是什么：采用宽松许可证；贡献入口、行为规范和安全说明都已公开。
- 最主要的公共入口在哪里：README 先说明项目定位，再直接链接贡献说明、语言包结构和自动检查入口。
- 当前维护状态如何：最近仍有版本发布，Issue 和 Pull Request 都能看到维护者持续回应。
- 最值得警惕的风险是什么：项目内容分散在多语言目录里，第一次进入时容易误判修改范围。
- 我的下一步动作是什么：尝试贡献，但先把候选对象收紧到单条文档修正或单语言示例补充。
```

### 2. 《公共入口缺口清单》片段

```md
| 对象 | 当前状态 | 为什么构成缺口 | 下一步建议 |
| --- | --- | --- | --- |
| 贡献入口 | README 已链接贡献说明，但缺少“第一次从哪里开始”的一句提示 | 外部人知道可以参与，但不知道第一步该看哪类对象 | 在 README 增补新贡献者入口句，并链接贡献说明中的对应段落 |
| 版本说明 | 已有 release，但变更摘要分散 | 外部人很难判断最近变化是否影响阅读和贡献选择 | 为最近版本补一段面向外部读者的变化摘要 |
```

### 3. 教师点评

- 这个样例的强项不在“写得多”，而在于所有对象都回到同一个项目级判断。
- 行动选择不是凭感觉跳出来的，而是被“入口清楚、维护存在、风险可控”三组证据支撑。
- 缺口清单没有停留在“缺文件”，而是解释了“为什么它阻断外部人进入”。

## 示例包 B：达标样例

### 1. 《项目阅读摘要》样例

```md
# 项目阅读摘要

- 项目主要是给命令行使用者提供示例。
- README、贡献说明和版本记录都能找到。
- 最近还有维护活动，所以项目不是完全停滞。
- 我认为它适合继续学习，必要时可以考虑后续尝试贡献。
```

### 2. 《公共入口缺口清单》片段

```md
| 对象 | 当前状态 | 为什么构成缺口 | 下一步建议 |
| --- | --- | --- | --- |
| README | 信息较多，但第一次进入时重点不够突出 | 外部人需要自己再找贡献入口 | 把项目定位和第一次进入路径再前置 |
```

### 3. 教师点评

- 这类样例已经能完成最小闭环，但仍偏“对象清单式”。
- 判断大体成立，不过还没有说清哪些对象真正改变了结论。
- 行动选择可以再收紧，避免“继续学习 / 尝试贡献”并列得过于宽泛。

## 示例包 C：常见问题样例

### 1. 《项目阅读摘要》样例

```md
# 项目阅读摘要

- 这是一个很热门的项目，star 很多，看起来很成熟。
- 仓库里文件很多，所以应该比较规范。
- 我准备直接尝试贡献。
```

### 2. 《公共入口缺口清单》片段

```md
| 对象 | 当前状态 | 为什么构成缺口 | 下一步建议 |
| --- | --- | --- | --- |
| README | 不够好 | 需要改进 | 优化 README |
```

### 3. 教师点评

- 这个样例的问题不是“判断方向一定错”，而是几乎没有证据链。
- `star` 数、文件数量和“应该比较规范”都不能替代真实的入口、治理和活动判断。
- 《公共入口缺口清单》没有解释缺口如何影响外部人进入，因此无法支撑后续行动选择。

## 快速提醒

- 不要把 star 数或名气直接等同于健康度。
- 不要把单个 `good first issue` 标签直接等同于高可参与性。
- 不要只记“仓库里有什么”，要写出“这些对象共同说明了什么”。
- 如果行动选择是“尝试贡献”，本章只需记下候选对象与判断理由，认领评论、提交信息和 PR 结构放到第 6 章再组织。
- 遇到 AI 项目时，要把权重、代码、数据说明与可重建性拆开判断。
