# 第二章《自由软件、开源软件与许可证》书稿提纲（草案）

> 当前状态：基于内部章节包与定向研究形成的书稿提纲草案。本文档用于进入正式章节写作前的结构收敛，不替代最终书稿正文。

## 开场段（草案）

一个项目团队把代码放到了 GitHub 上，仓库是公开的，README 也写得很完整，大家自然以为“这已经是一个开源项目了”。几周后，另一支队伍希望复用其中一部分代码，却发现仓库里没有 `LICENSE` 文件。代码明明看得见，为什么却不能理直气壮地拿来修改、再发布，甚至继续公开协作？这个看似细小的问题，恰好揭示了开源世界最容易被忽略的一层基础：开源并不只是把源代码公开出来，更是通过许可证把使用、修改、分发和再发布的边界明确写下来。本章要讨论的，正是自由软件、开源软件与许可证之间的关系，以及它们为什么构成现代开源开发的制度基础。

## 本章将回答的问题（Questions This Chapter Answers）

- 自由软件（Free Software）与开源软件（Open Source Software）到底是什么关系？
- 为什么“代码公开可见”并不自动等于“这是开源软件”？
- 为什么许可证（License）不是仓库里的装饰文件，而是开源协作的制度基础？
- MIT、BSD、Apache、GPL、LGPL、AGPL、MPL 这些常见许可证最值得区分的地方是什么？
- 当一个团队启动开源项目时，如何做出一个基本合理、可解释的许可证选择？

## 正文主线

本章建议采用 4 个一级小节。

## 1. 自由软件与开源软件，到底是什么关系

### 本节目标

建立两个核心概念的边界，避免读者把历史传统、价值诉求和制度表达混成一个模糊标签。

### 本节核心论点

- 自由软件首先强调 software freedom，而不是价格
- 开源软件强调基于许可证的开放协作与可复用性
- 两者有深刻连续性，但表达重点不同，不能简单当作完全同义词

### 本节建议写入的内容

- FSF 关于四项自由的基本表述
- OSI 关于 Open Source Definition 的基本判断标准
- 自由软件与开源软件的共同基础与不同重心

### 本节写法提醒

- 不把本节写成历史口水战
- 不把“自由”误写成“免费”

## 2. 为什么许可证是开源开发的制度基础

### 本节目标

让读者理解：开源并不是没有版权，而是通过版权与许可证建立清晰的协作边界。

### 本节核心论点

- 公开源码只是技术状态，许可证才是制度安排
- 没有许可证时，默认版权保护仍然存在
- 开源协作之所以可持续，是因为权利与义务被提前写明

### 本节建议写入的内容

- “No License” 场景
- 版权、分发、衍生作品、Notice、专利授权的最低限度解释
- 为什么许可证帮助陌生人之间展开协作

### 本节不展开的内容

- 法律实务细节
- 复杂诉讼案例

## 3. 常见许可证如何区分

### 本节目标

用最少但稳定的区分框架，帮助读者建立对常见许可证的第一层认知。

### 本节核心论点

- 许可证首先可按宽松式（Permissive）、弱 Copyleft、强 Copyleft 三大类理解
- MIT / BSD / Apache 都较宽松，但 Apache 对专利与 NOTICE 更重视
- GPL / LGPL / AGPL / MPL 的关键差异在 Copyleft 作用范围，而不是口号化标签

### 本节建议写入的内容

- MIT 与 BSD-3-Clause 的最小差异
- Apache-2.0 的专利与 NOTICE 特征
- GPL、LGPL、AGPL、MPL-2.0 的最小稳定区分

### 本节写法提醒

- 不逐条解释长许可证文本
- 不把兼容性矩阵塞进正文

## 4. 如何为一个项目做出一个可解释的许可证选择

### 本节目标

让本章直接服务真实项目场景，使读者知道“理解许可证”最终要落到真实决策上。

### 本节核心论点

- 许可证选择不是寻找“最正确答案”，而是对项目目标、协作方式和复用边界做出清晰说明
- 项目发起者最重要的是做出可解释、可执行、与项目现实相匹配的选择
- 对第三方代码、依赖、素材和模型的处理，应比“自己写的代码”更谨慎

### 本节建议写入的内容

- 项目许可证选择的四五个判断维度
- 宽松式与 Copyleft 的典型适用场景
- 何时需要暂停决定并先清理第三方来源

## 本章小结（预期方向）

本章的小结应明确指出：自由软件和开源软件都在回应软件使用、修改和传播的控制权问题，但现代开源协作之所以能稳定运行，并不是因为大家天然彼此信任，而是因为许可证把权利与义务事先写清楚了。理解了这一点，读者才会知道为什么开源项目需要 `LICENSE`，为什么不同许可证会影响协作边界，以及为什么许可证选择是项目设计的一部分，而不是发布前随手补上的文件。
