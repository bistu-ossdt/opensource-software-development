# 第 6 章 案例教学包

## 对应章节与用途

- 对应书稿：[06-participating-in-open-source-projects.md](06-participating-in-open-source-projects.md)
- 配套学习指南：[chapter-06-study-guide.md](chapter-06-study-guide.md)
- 配套教学指南：[chapter-06-instructor-guide.md](chapter-06-instructor-guide.md)
- 本案例包用于帮助学生把“第一次贡献”理解为一条真实进入路径，而不是把仓库地址清单误当成参与方法。

## 主案例与辅助案例

| 案例 | 本章主要说明什么 | 推荐观察对象 |
| --- | --- | --- |
| tldr-pages / tldr | 新贡献者友好的第一次贡献路径 | `CONTRIBUTING.md`、文档规范、轻量 PR 路径 |
| CPython 或 VS Code | 成熟项目中的 PR、Checks、review 与边界控制 | `Issue`、PR 描述、Checks、review 往返 |
| Linux | 重要项目不等于适合作为第一次进入对象 | 贡献门槛、维护层级、长期上下文密度 |
| OpenClaw | AI 原生项目中的第一次贡献入口不只等于“训练模型” | 模型卡、评测说明、部署脚本、来源说明 |

## 推荐课堂使用方式

- 先用书稿中的 `DATA_DIR` 贯穿案例，把“边界判断 -> 认领 -> 提交 -> PR -> checks -> review -> merge”这条抽象链讲清。
- 再用 `tldr` 说明一个新贡献者友好项目的最小进入路径，让学生看到“第一次贡献”可以从轻量但真实的对象开始。
- 再用 CPython 或 VS Code 说明成熟项目里为什么 PR 描述、Checks 和 review 回复缺一不可。
- 用 Linux 做反衬，说明“著名、重要、影响大”不等于“适合第一次进入”。
- 最后用 OpenClaw 说明 AI 原生项目的第一次贡献也可以从文档、评测、部署与说明对象切入。

## 与书稿贯穿案例的配合

- `DATA_DIR` 用于讲清抽象方法，帮助学生先在一个收束的对象上看懂第一次贡献的最小链条。
- `tldr` 用于把这条最小链条映射到一个新贡献者更容易进入的真实仓库。
- CPython 或 VS Code 用于展示成熟项目里“说明、检查、评审”如何共同构成贡献对象。
- Linux 用于提供反例意识：重要项目不一定适合第一次进入。
- OpenClaw 用于说明 AI 原生项目的第一次贡献也仍然遵守公共记录、验证与评审逻辑。

## 推荐讨论问题

- 为什么一个项目越重要，反而越不一定适合作为第一次贡献入口？
- 新手友好标签为什么只能算入口信号，而不是接收承诺？
- 什么样的 PR 描述能把一次修改真正变成“可判断对象”？
- AI 项目的第一次贡献为什么不一定先从模型训练或核心推理代码开始？

## 与团队项目主线的连接

- 团队可用 `tldr` 的路径设计反过来审视自己的项目：第一次进入是否真的足够清楚。
- 团队可借 CPython 或 VS Code 的 PR 组织方式，补齐自己仓库中的说明与验证结构。
- 团队可借 Linux 的反例意识避免把“困难对象”误当成“更有价值的首次尝试”。
- 团队可借 OpenClaw 的案例意识，为自己的 AI 相关部分设计更清楚的文档、评测和部署入口。

## 使用边界

- 不要把案例包讲成“推荐仓库清单”。
- 不要只比较 star 数或名气，而忽略第一次进入需要的上下文密度。
- 不要跳过书稿中的 `DATA_DIR` 贯穿案例，直接把真实仓库案例讲成平台操作展示。
- 不要把 AI 项目的参与路径窄化成模型训练或核心算法修改。
