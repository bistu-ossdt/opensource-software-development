# 历史三个实践与当前课程设计适配性分析参考

## 目的

本文档分析历史课程中的三个实践任务，判断它们是否适合纳入当前课程设计，以及如果保留，应放入当前课程材料体系的哪个部分。

这里分析的不是“这些实践当年是否有效”，而是它们在当前课程定位下是否仍然成立，以及需要如何升级。

## 分析对象

本次分析基于以下三个历史实践文档：

- `ref/开源软件开发技术课程实践1-Blog.md`
- `ref/开源软件开发技术课程实践2-项目分析.md`
- `ref/开源软件开发技术课程实践3-项目建设.md`

分析时同时参照了当前课程设计原则、实验蓝图和课程结构规范。

## 总体判断

这三个实践的历史方向总体是对的，而且能看出课程长期以来一直不是停留在概念教学，而是强调真实动手。

如果把它们抽象成实践骨架，可以概括为：

1. 使用开源工具完成一个真实小任务
2. 阅读并分析一个真实开源项目
3. 组队建设并发布一个开源项目

这个骨架与当前课程定位并不冲突，反而很有价值。

但在当前课程设计下，它们 **不适合原样保留**。问题不在于方向错误，而在于：

- 技术细节明显过时
- 对现代 GitHub 开源协作的覆盖不足
- 对治理、CI、测试、安全、版本与发布的要求不够
- 部分内容更适合作为实验材料或教学支持，而不是教材正文

因此，结论是：

- 三个实践都可以保留
- 但必须按照当前课程结构重新分层和重写
- 其中实践 2 和实践 3 应保留为核心实践
- 实践 1 适合降级为前置热身实践或并入其他实践的子任务

## 分项分析

## 实践 1：使用开源软件建立博客

### 历史价值

这个实践主要训练：

- 阅读安装文档
- 安装与配置工具
- 使用命令行
- 解决环境问题
- 初步使用 Git 与 GitHub
- 完成一次公开发布

这些能力在当前课程里依然重要，因此它不是“无用实践”。

### 主要问题

它的问题主要在于强绑定某一套旧技术路径：

- 强绑定 Hugo 和 GitHub Pages
- 含有明显时代性的操作细节
- 包含不应鼓励沿用的过时技巧，例如关闭 SSL 校验
- 更偏“使用开源软件”而不是“进行开源软件开发”

### 当前适配性判断

适合保留其“热身价值”，但不适合继续作为课程三大核心实践之一。

### 在当前课程中的推荐位置

- `Lab / Project Materials`
  作为前置热身实践，训练环境搭建、文档阅读、简单发布
- `Student Study Guide`
  可作为“第一次通过文档解决问题”的学习建议案例
- `Instructor Guide`
  可作为低门槛入门活动或可选加分任务

### 不建议放入的部分

- 不建议进入 `Technical Book Manuscript` 主体
- 不建议继续作为课程主线项目

### 现代化改造建议

更适合把它改造成以下类型之一：

- 开源工具上手与环境排错热身实验
- 项目文档站点 / 项目主页发布子任务
- 团队项目中的文档发布和项目展示子任务

## 实践 2：开源项目分析

### 历史价值

这个实践与当前课程设计高度契合。它训练的是：

- 选择一个真实开源项目
- 阅读项目资料
- 分析项目功能、历史、组织与结构
- 学会从仓库外观走向项目理解

这正好对应当前课程中“阅读与理解开源项目”的核心目标。

### 主要问题

它的分析框架偏旧，明显更适合早期 SourceForge / Apache 式项目，而不是今天的 GitHub 为中心的现代开源项目。

例如，它较强调：

- 邮件列表
- 维基
- 缺陷追踪系统

但对现代项目更关键的部分覆盖不足，例如：

- `README`
- `LICENSE`
- `CONTRIBUTING.md`
- `CODE_OF_CONDUCT.md`
- `SECURITY.md`
- Issues / Pull Requests
- Releases
- CI workflows
- Discussions
- 治理与维护者结构

### 当前适配性判断

非常适合保留，而且应保留为当前课程的核心前期实践之一。

### 在当前课程中的推荐位置

- `Lab / Project Materials`
  作为前期必做实验
- `Student Study Guide`
  作为“如何读一个开源项目”的学习指导
- `Instructor Guide`
  作为项目分析的 rubric、选题建议、案例选择建议
- `Case Library`
  可沉淀优秀项目分析样本与对照案例

### 与课程章节的对应关系

主要对应：

- 第 3 章 开源社区与治理
- 第 5 章 阅读与理解开源项目
- 第 6 章 参与开源项目

### 现代化改造建议

应把旧分析框架升级为现代 GitHub 项目分析框架，至少补入：

- 仓库首页与关键文件分析
- 治理与贡献入口分析
- Issue / PR / Release / CI 观察
- 社区活跃度与维护状态判断
- “我可以从哪里开始贡献”的分析

## 实践 3：开源项目建设发布

### 历史价值

这是三个实践里最重要的一个，也是和当前课程最匹配的一个。它证明课程长期以来一直强调：

- 团队协作
- 开源项目立项
- 许可证选择
- 文档准备
- GitHub 公开发布

这与当前课程“团队开源项目主线”的方向完全一致。

### 主要问题

它的问题不是方向，而是基线过旧。旧模板较强调：

- `README`
- `INSTALL`
- `FAQ`
- `CREDITS`
- `HISTORY`
- `LICENSE`
- Wiki
- 邮件列表
- 缺陷追踪系统

但没有系统要求现代开源项目更关键的能力证据，例如：

- Issue 拆分与项目看板
- Pull Request 协作
- Code Review 证据
- 测试与 CI
- 版本策略与 `CHANGELOG`
- `CONTRIBUTING.md`
- `CODE_OF_CONDUCT.md`
- `SECURITY.md`
- `CODEOWNERS`
- 发布页面与 release evidence
- 依赖治理与基本安全意识
- AI 辅助开发记录与人工验收证据

### 当前适配性判断

必须保留，而且应升级为当前课程的主线实践。

### 在当前课程中的推荐位置

- `Lab / Project Materials`
  作为学期主线实践
- `Course Syllabus / Course Guide`
  作为课程项目里程碑与阶段安排
- `Student Study Guide`
  作为“如何把每章内容落实到团队项目”的学习支持
- `Instructor Guide`
  作为项目组织、评价和复盘的核心支持材料
- `Case Library`
  可沉淀优秀学生项目仓库样本

### 与课程章节的对应关系

主要对应：

- 第 2 章 自由软件、开源软件与许可证
- 第 3 章 开源社区与治理
- 第 4 章 开源开发的基本工程流程
- 第 6 章 参与开源项目
- 第 8 章 组织与发布你的开源项目

### 现代化改造建议

应把它升级为分阶段团队项目，而不是一次性提交的“大作业”，例如分为：

- 项目立项与许可证选择
- 治理基线建立
- GitHub 协作与最小 CI
- MVP 迭代
- AI 辅助任务与人工治理
- 发布、归档与项目复盘

## 三个实践在当前课程中的结构性放置

| 历史实践 | 是否保留 | 当前角色 | 主要放置层 |
| --- | --- | --- | --- |
| 实践 1 Blog | 保留，但降级 | 前置热身实践 / 发布子任务 | `Lab / Project Materials`，少量进入 `Student Study Guide` / `Instructor Guide` |
| 实践 2 项目分析 | 保留并升级 | 前期核心实践 | `Lab / Project Materials` + `Student Study Guide` + `Instructor Guide` + `Case Library` |
| 实践 3 项目建设 | 保留并升级 | 学期主线实践 | `Lab / Project Materials` + `Course Syllabus / Course Guide` + `Student Study Guide` + `Instructor Guide` + `Case Library` |

## 对当前课程设计的直接结论

如果要保留“三个实践”这一历史结构，更适合重构为：

1. 开源项目阅读与分析
2. 团队项目初始化与治理建立
3. 团队开源项目建设、发布与复盘

其中，原来的 Blog 实践不应再作为独立核心实践，而应转化为：

- 课程前置热身实践
- 或团队项目中的项目主页 / 文档发布子任务

这意味着，历史三个实践不是被否定，而是被吸收到当前更成熟的课程结构之中。
