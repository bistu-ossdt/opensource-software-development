> 当前状态：Round 1 骨架草案。本文档用于第三章的结构设计、旧材料迁移和后续定向研究，不是书稿正文。

## 章节信息

- 章节编号：第 3 章

- 英文标题：`Open Source Communities and Governance`

## 本章定位

本章承接前两章，从“历史形成”和“制度边界”继续推进到“开放协作如何被稳定组织起来”。如果说第 1 章回答的是“开源如何形成”，第 2 章回答的是“开源如何获得制度基础”，那么第 3 章要回答的是“开源社区如何持续运作”。

本章主要解决五个问题：

1. 为什么开源不是“公开仓库 + 接收代码”就能自然成立
2. 社区、贡献者、维护者、核心团队、基金会等角色之间是什么关系
3. 一个项目的讨论路径、贡献路径和决策路径通常如何组织
4. 为什么 `CONTRIBUTING.md`、`CODE_OF_CONDUCT.md`、Issue / PR 模板等文件属于治理基础设施
5. 一个项目如何从“几个人一起写代码”走向“可持续的公共协作系统”

本章不承担 GitHub Flow、Pull Request 操作、CI 配置、首次贡献步骤或项目初始化细节。这些内容分别在第 4、6、8 章展开。

本章与前后章节的关系：

- 与第 1 章的关系：第 1 章说明开源运动与生态如何形成，本章说明这些历史传统在真实项目中如何演化为社区结构与治理机制
- 与第 2 章的关系：第 2 章处理“权利与义务”的制度基础，本章处理“角色与流程”的协作基础
- 与第 4 章的关系：本章解释“为什么需要治理与规则”，第 4 章解释“这些规则如何通过工程流程执行”
- 与第 5 章的关系：本章建立阅读治理文件和识别社区结构的能力，第 5 章再把这些能力整合为项目级判断
- 与第 6 章的关系：本章解决“社区怎样运作”，第 6 章解决“个人如何实际进入并持续参与”
- 与第 8 章的关系：本章面向读者理解治理，第 8 章面向创建者设计治理基线

## 本章学习目标（Learning Objectives）

学完本章后，学生原则上应能够：

- 解释为什么开源协作需要社区治理，而不只是代码托管平台
- 识别使用者、贡献者、评审者、维护者、核心团队等常见角色及其基本关系
- 读懂一个项目中常见的治理入口，如 `README`、`CONTRIBUTING.md`、`CODE_OF_CONDUCT.md`、Issue / PR 模板、`GOVERNANCE.md`
- 说明开源项目中讨论、决策、评审和冲突处理为什么应尽量公开、异步、可归档
- 对一个真实项目做出初步的社区结构与治理成熟度判断

## 正文主叙述结构（骨架版）

建议第三章正文采用 4 个一级小节。

### Opening / 开场切入

建议的开场问题：

- 为什么有些公开仓库看起来“能看代码”，却很难真正参与？
- 为什么一个项目要想长期吸引贡献者，必须把角色、规则、沟通方式和行为边界写清楚？

可选开场场景：

- 场景 A：一个仓库代码很多，但没有清晰的贡献入口和行为规范，新人不知道如何进入
- 场景 B：两个经典项目都能长期运行，但它们的治理结构和决策方式并不相同，为什么仍都离不开治理

当前建议：优先使用场景 B，以 `Linux` 和 `Apache` 做轻量对照，快速建立“维护者主线”和“社区化治理”两种经典路径的视角。

### 第一节：开源社区不是代码的附属物

本节要解决的问题：

- 社区和仓库是什么关系
- 为什么“大教堂 / 市集”仍有解释价值，但不足以覆盖现代治理现实
- 为什么开源协作必须依赖公共规则、公共记录和可信的角色结构

应吸收的旧材料：

- `03-development-models-and-characteristics.md` 中关于市集模式、角色灵活性、邮件列表文化的内容
- `250622 第 4 部分 PPT` 中关于组织和模式的讲授素材

风险控制：

- 不把“市集模型”讲成今天开源治理的完整解释
- 不把社区浪漫化为“自然形成的自由合作”

### 第二节：角色、权力与责任如何被组织起来

本节要解决的问题：

- 使用者、贡献者、评审者、维护者、核心团队之间的差异是什么
- 为什么现代项目需要清晰的角色梯度和贡献路径
- BDFL、PMC、委员会、基金会等治理形态分别解决什么问题

应吸收的旧材料：

- `03-development-models-and-characteristics.md` 中关于角色分工和组织结构的内容
- `04-participating-in-and-organizing-open-source-projects.md` 中少量关于组织和角色的内容
- 少量 `05-leveraging-open-source-software-resources.md` 中关于基金会和组织支持的素材

风险控制：

- 不把 Apache 模式误写成全部项目的标准答案
- 不把“有没有基金会”误当作治理成熟度的唯一标志

### 第三节：讨论、贡献与决策为什么要公开、异步、可归档

本节要解决的问题：

- 为什么开源社区重视公共讨论和可追溯记录
- 邮件列表、论坛、Discussions、Issue、PR 评论等媒介分别承担什么功能
- 为什么 `CONTRIBUTING.md`、Issue / PR 模板、`CODE_OF_CONDUCT.md` 是治理入口而不只是形式文件

应吸收的旧材料：

- `03-development-models-and-characteristics.md` 中关于邮件列表、论坛、沟通礼仪的内容
- `250622 第 5 部分 PPT` 中与参与和社区礼仪相关的素材
- `open-source-community-governance-best-practices-reference.md` 中关于治理文档和公开沟通的提炼

风险控制：

- 不把本节写成 GitHub 功能说明书
- 不把行为规范讲成抽象口号，而要落到协作门槛和冲突处理

### 第四节：从治理文件到社区健康，项目如何保持可持续

本节要解决的问题：

- 为什么社区健康、维护者负担和继任问题属于治理核心
- 为什么现代项目要关注贡献者留存、响应速度和角色集中度
- 为什么企业参与、基金会支持和维护者可持续性需要透明规则

应吸收的旧材料：

- 旧材料中关于赞助、组织支持和社区长期发展的内容
- `open-source-community-governance-best-practices-reference.md` 中关于社区健康、维护者可持续性和企业参与透明度的内容

本节收束目标：

- 把“社区治理”落为开源开发的基础设施
- 自然引到第 4 章的工程流程和第 5 章的项目评估

## 本章的明确取舍

### 直接保留的内容骨架

- 社区和治理是开源协作的基础，而不是外围文化
- 角色分工、维护者体系和公共沟通的重要性
- 邮件列表 / 论坛 / 讨论入口作为可归档协作介质
- 社区健康、维护者负担与长期可持续性的基本判断

### 必须重写的地方

- 旧稿中把“大教堂 / 市集”当作总解释框架的部分
- 旧稿中组织结构过于静态、单一的部分
- 对邮件列表和现代平台关系的表述，要从“旧工具介绍”改写为“治理媒介演化”
- 对赞助、企业参与和社区支持的表述，要从宣传口吻改写为治理口吻

### 不进入本章正文的旧内容

- GitHub 注册、安装、分支和 PR 操作步骤
- 生命周期、设计、实现、测试、编码规范等工程实现细节

## Study Guide Mapping

本章对应的 `Student Study Guide` 应至少覆盖：

- 阅读顺序建议：先理解“为什么需要治理”，再理解角色与路径，最后识别仓库中的治理入口
- 常见误区：
  - 把社区理解成“微信群聊”或“粉丝群”
  - 认为治理只对大型项目重要
  - 认为行为准则只是形式文件
  - 把项目是否上 GitHub 等同于社区是否开放
- 与团队项目主线的连接提醒：
  - 团队项目从这一章开始要建立最小治理基线，而不只是分工写代码
  - 团队需要明确角色、贡献入口和基本协作规则
- 自查问题：
  - 你能否从一个仓库中识别出治理入口和贡献路径
  - 你能否解释维护者为什么不仅是“会合并代码的人”

## Instructor Guide Mapping

本章对应的 `Instructor Guide` 应至少覆盖：

- 教学重点：
  - 治理是开源开发技术的组成部分，而不是软性附属话题
  - 要把角色、规则、媒介和健康度放在一起讲，而不是分散罗列
  - 本章应与团队项目治理基线建立直接连接
- 易错点：
  - 把“大教堂 / 市集”讲成今天治理的完整答案
  - 把 Linux 的治理方式误当成所有项目的模板
  - 把 GitHub 社区功能误当成治理本身
- 适合课堂讨论的问题：
  - 一个项目为什么需要把规则写下来，而不是靠“大家都懂”
  - 对学生团队项目来说，最小治理集合应该包括什么
  - 公司参与是增强还是削弱开源社区，取决于哪些治理条件
