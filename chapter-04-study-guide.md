# Chapter 4 Student Study Guide

## 对应书稿

- 对应章节：[04-participating-in-and-organizing-open-source-projects.md](04-participating-in-and-organizing-open-source-projects.md)
- 章节主题：开源开发的基本工程流程
- 使用目标：帮助学生把现代开源开发理解为围绕变更控制组织起来的工程系统

## 如何使用本章

- 先带着“为什么公开协作反而不允许直接改主线”进入本章。
- 阅读时把 `Issue`、`Branch`、`Commit`、`Pull Request`、`Review`、`Checks`、`Merge`、`Release` 视为同一条链上的对象，而不是独立工具名词。
- 配合图 4-1 与表 4-1，把每个对象放回“它回答什么问题”。
- 每读完一节，都问自己：这一步在保护什么，降低了什么风险。

## 本章学习目标

- 能解释工程流程为什么不是官僚负担，而是变更控制。
- 能说明 `Issue`、`Branch`、`Commit` 在流程中的不同作用。
- 能解释 Pull Request、Review、Checks 与 Merge 如何共同构成门禁链条。
- 能说明测试、CI、依赖检查、密钥保护为什么属于日常流程的一部分。
- 能区分“可合并”与“可发布”。

## 建议阅读顺序

1. 先读第 1 节，建立“公共主线必须可被信任”的前提。
2. 再读第 2 节，用工作项、分支和提交理解“变更边界”。
3. 接着读第 3 节，看 Pull Request 如何把修改变成可判断对象。
4. 再读第 4 节，理解测试、CI 与自动化门禁为什么进入主流程。
5. 最后读第 5 节，理解依赖、密钥与发布完整性为什么不是外围小事。

## 关键术语与易混点

- `Issue`：把问题、需求或任务变成可追踪对象。
- `Branch`：为一组修改建立隔离边界。
- `Commit`：最小历史单元，不是随手保存按钮。
- `Pull Request`：把变更包装成公共判断对象。
- `Review`：人工判断，负责边界、设计与质量。
- Checks / CI：自动化验证，不替代人工判断，但构成最低门禁。
- `dependency review`：检查依赖变化是否引入风险。
- `secret scanning / push protection`：在流程中前移凭据泄露检查。

## 常见误区

- 流程越多，说明项目越不开放。
- 分支只是为了备份代码。
- Pull Request 只是“代码提交入口”，与解释修改无关。
- 测试通过就等于可以发布。
- 依赖风险、密钥泄露和自动化权限属于安全组的外围工作。

## 如果时间有限，最少应抓住什么

- 现代开源开发的核心是变更控制，而不是直接改主线。
- `Issue -> Branch -> Commit -> Pull Request -> Review -> Checks -> Merge -> Release` 是同一条工程链。
- “可以合并”和“准备发布”之间始终还有一道额外门槛。

## 本章最容易读偏的地方

- 把本章读成 GitHub 按钮教程，而不是流程抽象。
- 把 CI 看成“自动跑一下测试”，忽略其门禁与一致性作用。
- 把发布看成打标签动作，而忽略版本边界与对外承诺。

## 阅读检查点

- 能否说明为什么默认分支必须保持可被信任。
- 能否区分 `Issue`、`Branch`、`Commit`、`Pull Request` 在流程中的职责。
- 能否解释 Review 与 Checks 为什么都不可缺。
- 能否说清“合入主线”与“发布版本”之间为什么还有额外门槛。
- 能否把依赖审查、密钥保护和发布说明放回同一条变更控制逻辑中。

## 与团队项目主线的连接

- 本章把第 3 章中的角色与治理结构推进为“团队项目如何把修改安全地进入主线”。
- 团队项目应在这一章之后明确基本工作流：工作项、分支命名、提交粒度、PR 规则与评审方式。
- 团队应建立最小 CI 和合并前检查习惯。

## 完成本章后应留下的项目产物

- 形成一份《最小工作流与评审/发布清单》。
- 写明从 `Issue` 到 `Release` 的基本流程。
- 列出合并前的最小评审与检查规则。
- 说明发布时需要补齐的版本与说明对象。

## 建议的课后巩固任务

- 为团队项目完成《最小工作流与评审/发布清单》初稿。
- 读一个真实 PR，判断其中哪些信息让这组修改变得“可判断”。
- 用一个真实仓库说明其工作流、Review 和 Checks 是如何共同构成门禁的。

## 与前后章的衔接

- 本章承接第 3 章的治理结构，回答“修改如何进入项目”。
- 下一章会进一步说明：面对一个陌生仓库时，如何综合许可证、治理和流程判断其可参与性与可复用性。
