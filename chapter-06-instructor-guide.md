# 第 6 章 教学指南

## 对应书稿与使用方式

- 对应章节：[06-participating-in-open-source-projects.md](06-participating-in-open-source-projects.md)
- 建议用途：课程后半部分进入真实参与路径的核心单元
- 默认按一个 `100` 分钟教学单元设计；如需拆分，可分为“第一次贡献入口”与“PR / review / 持续参与”两次 `50` 分钟教学
- 当前状态：骨架版；后续将随书稿扩写同步细化

## 教学定位

本章不是“GitHub 提交技巧课”，而是帮助学生理解：一次真正的开源参与，必须在别人的维护责任之内，把修改组织成项目能够理解和接受的对象。

## 本章教学目标

- 让学生理解第一次参与首先考验边界感，而不是编程炫技。
- 让学生理解第一次贡献为何适合从小而清楚的任务开始。
- 让学生理解 PR 描述、任务说明和 review 回复同样属于贡献本体。
- 让学生理解持续参与来自稳定协作，而不是偶然一次合并。

## 本章与前后章的教学边界

- 本章承接第 5 章的项目阅读，讨论“看懂之后如何进入”。
- 本章不展开 AI 工具使用与 agentic workflow，那是第 7 章的重点。
- 本章重点是参与路径和协作纪律，而不是复杂实现难题。

## 教学重点

- 第 1 节必须让学生建立“项目需要什么”优先于“我想改什么”的意识。
- 第 2 节要讲透为什么第一次贡献更适合清楚边界任务。
- 第 3 节必须把贡献说明讲成贡献对象的一部分。
- 第 4 节要把 review 从“打分”改讲成“协作延续”。

## 易错点与难点

- 学生容易把参与理解成个人能力展示。
- 学生容易忽视任务范围控制与边界判断。
- 学生容易觉得 review 是对人的否定。
- 学生容易把一次合并成功误解为持续参与已经自然形成。

## 课堂上不应展开过深的内容

- 不应把本章讲成“如何刷开源经历”的短期策略课。
- 不应耗费大量时间在 Git 命令细节和平台按钮演示上。
- 不应把课堂主线变成复杂代码实现案例拆解。

## 推荐课堂组织

- `10` 分钟：从“为什么很多人会写代码，却做不好第一次贡献”开场。
- `20` 分钟：讲第 1 节，建立边界优先意识。
- `20` 分钟：讲第 2 节，比较文档、测试、小修复与大功能的不同入口风险。
- `25` 分钟：讲第 3-4 节，分析一个真实 PR 的说明与 review 往返。
- `15` 分钟：讲第 5 节，把一次贡献放回持续参与。
- `10` 分钟：让学生为团队项目写出一份“外部人第一次贡献如何进入”的说明。

## 推荐讨论问题

- 为什么“代码写得出来”并不等于“贡献一定会被接受”？
- 什么样的任务最适合作为第一次贡献？
- PR 描述为什么不应被视为附属材料？
- 维护者为什么常常在 review 中强调范围控制、测试和长期维护成本？

## 案例使用建议

- 适合使用一个真实的初次贡献 PR 作为主案例。
- 如果使用团队项目案例，可让学生以“外部贡献者”身份审视自己项目。
- 案例重点应放在边界、说明、review 和修改收敛，而不是代码量。

## 与团队项目主线的阶段映射

- 本章要求团队开始设计“别人如何参与我们的项目”。
- 本章结束后，团队至少应有第一次贡献入口、最小贡献说明和最小 review 规则。
- 本章对应的阶段产物是《第一次贡献路径说明》。

## 可评价证据

- 一个真实项目的首次贡献路径分析。
- 团队项目的首次贡献说明与任务入口设计。
- 一份 PR 描述样稿与 review 回复样稿。

## 最小达标证据 / 提升证据

- 最小达标证据：能写出团队项目的第一次贡献路径，并解释一个 PR 需要说明哪些核心信息。
- 提升证据：能结合真实项目分析 review 往返过程，并将其迁移到团队项目的贡献规范中。

## 基线层与扩展层建议

- 基线层：项目边界、第一次贡献入口、贡献说明、PR 组织、review 回复、持续参与。
- 扩展层：更复杂的贡献拆分、长期 issue 跟进、权限演化、共同维护与新人引导。

## 教学结构参考

以下参考用于借鉴教学结构与支架方法，不用于照搬内容或作业设计。

- [Open Source Guides](https://opensource.guide/zh-cn/how-to-contribute/): 借鉴其把第一次参与拆成“先观察、再进入、再协作”的组织方式。
- [Cornell CS5152](https://www.cs.cornell.edu/courses/cs5152/2019fa/): 借鉴其围绕真实开源项目组织团队参与与 mentor 支撑的项目式做法。
- [Producing Open Source Software](https://producingoss.com/): 借鉴其把贡献、沟通、评审与社区参与写成可复用方法的结构。
