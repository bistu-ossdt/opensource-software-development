# 第 6 章 教学指南

## 对应书稿与使用方式

- 对应章节：[06-participating-in-open-source-projects.md](06-participating-in-open-source-projects.md)
- 建议用途：课程后半部分进入真实贡献路径的核心单元
- 默认按一个 `100` 分钟教学单元设计；如需拆分，可分为“第一次贡献入口”与“PR / review / 持续参与”两次 `50` 分钟教学
- 当前状态：成熟教学包；已补齐评分尺与示例包，可直接用于教学、批改与课堂校准

## 教学定位

本章不是“GitHub 提交技巧课”，也不是“如何快速刷一次开源经历”的策略课。本章要帮助学生理解：一次真正的开源参与，必须在别人的维护责任之内，把修改组织成项目能够理解、评审和接受的对象。

## 本章教学目标

- 让学生理解第一次参与首先考验边界感，而不是编程炫技。
- 让学生理解为什么第一次贡献更适合从小而清楚、可验证的任务开始。
- 让学生理解认领、commit message、PR 描述和 review 回复都属于贡献本体。
- 让学生理解持续参与来自稳定协作，而不是偶然一次合并。
- 让学生理解 AI 进入开发之后，这条贡献路径并不会消失。

## 本章与前后章的教学边界

- 本章承接第 5 章的项目阅读，讨论“看懂之后如何进入”。
- 本章重点是参与路径和协作纪律，而不是复杂实现难题。
- 本章不展开 AI 工具使用、agentic workflow 或模型开发细节，那是第 7 章的重点。

## 教学重点

- 第 1 节必须让学生建立“项目需要什么”优先于“我想改什么”的意识。
- 表 6-1 应作为课堂上的第一次任务筛选框架，而不是读过即止的附表。
- 图 6-1 应作为本章结构骨架，帮助学生把 Issue、分支、commit、PR、review、checks、merge 视作同一条链。
- `DATA_DIR` 贯穿案例要讲成完整闭环：为什么适合、如何认领、如何写说明、如何通过检查、如何回应 review、如何收束到 merge。

## 易错点与难点

- 学生容易把参与理解成个人能力展示，而不是公共协作。
- 学生容易忽视“问题存在”与“项目当前愿意这样解决”之间的差别。
- 学生容易把标签当作直接命令，而不再核查上下文是否已经变化。
- 学生容易觉得 review 是对人的否定，而不是项目把上下文重新传回来的过程。

## 课堂上不应展开过深的内容

- 不应把课堂主线变成 Git 命令细节或平台按钮演示。
- 不应把时间耗在复杂代码实现案例拆解上。
- 不应提前进入第 7 章的 AI 工具操作和 agent 配置细节。

## 推荐课堂组织

- `10` 分钟：从“为什么很多人会写代码，却做不好第一次贡献”开场。
- `20` 分钟：讲第 1 节，让学生学会判断“问题是否仍然活跃、是否适合当前版本窗口”。
- `20` 分钟：讲第 2 节和表 6-1，比较文档、测试、小修复与大功能入口的差异。
- `25` 分钟：讲第 3 节和图 6-1，用 `DATA_DIR` 案例串起一次最小贡献链。
- `15` 分钟：讲第 4 节，重点做一次 review 线程角色扮演或讨论。
- `10` 分钟：讲第 5 节，把一次贡献放回持续参与和角色梯度。

## 推荐课堂活动

- 活动 1：给学生两个看起来都能做的问题，让他们用第 1 节的方法判断哪一个更适合作为第一次贡献。
- 活动 2：让学生把 `DATA_DIR` 案例写成一条认领评论、一条 commit message 和一份最小 PR 描述，并说明“故意没改什么”。
- 活动 3：分组扮演 Reviewer 与 Contributor，围绕一个 `request changes` 线程继续往返，要求学生只在同一条 PR 上推进，并说明哪些检查结果必须重新确认。
- 活动 4：让团队从“外部贡献者”的视角，重新审视自己的项目目前是否真的具备第一次贡献入口。

## 推荐讨论问题

- 为什么“代码写得出来”并不等于“贡献一定会被接受”？
- 什么样的任务最适合作为第一次贡献？为什么？
- 为什么 `good first issue` 和 `help wanted` 只能算入口信号，不能算维护承诺？
- PR 描述为什么不应被视为附属材料？
- 一次贡献被拒绝后，什么样的后续动作才是成熟的？

## 案例使用建议

- 优先使用书稿中的 `DATA_DIR` 贯穿案例，因为它已经完整覆盖了边界判断、说明、review 和收束。
- 如需转入真实仓库案例，先让学生把 `DATA_DIR` 例子中的判断链说清，再进入本章案例教学包中的 `tldr`、CPython / VS Code、Linux 与 OpenClaw。
- 如使用真实项目案例，优先选择公开记录完整、第一次贡献路径清楚的仓库。
- 如果使用团队项目案例，应强制学生以“外部贡献者”身份来审视自己项目，而不是以项目成员身份自说自话。

## 与团队项目主线的阶段映射

- 本章要求团队开始设计“别人如何参与我们的项目”。
- 本章结束后，团队至少应有：第一次贡献入口、最小贡献说明、最小 review 回复规则。
- 本章对应的阶段产物是《第一次贡献路径说明》、一组第一次贡献候选任务，以及一套真实或模拟的贡献包。

## 评分尺使用建议

- 本章评分以 `chapter-06-lab-project-materials.md` 中的实验级评分尺为准，按 `5` 个维度、每项 `20` 分组织。
- 建议评分顺序固定为：先看《第一次贡献目标分析》，再看贡献包完整度与说明质量，最后看团队项目的外部入口设计。
- 不要把“是否已被 merge”当作直接加分项；核心看判断质量、公共记录组织质量、验证质量和 review 协作能力。
- 如果是真实贡献但尚未收到维护者回复，不应因此直接扣分；可依据学生补充的 PR 描述、验证说明和预备 review 回复判断其方法是否成立。
- 若缺少《第一次贡献目标分析》、完整贡献包、或团队项目入口设计三项中的任一项，原则上不判为“完整达标”。

## 示例包与课堂校准

- 附录支撑中的示例包固定分为“高质量样例 / 达标样例 / 常见问题样例”三组，建议按这个顺序做课堂校准。
- 更稳妥的做法是：先让学生独立完成自己的第一次贡献包，再用“常见问题样例”做集体挑错，再对照“达标样例”校准最低通过线，最后再展示“高质量样例”。
- 样例包全部围绕同一类轻量外部贡献对象展开，目的是校准贡献链质量，而不是引入新的项目复杂度。
- 如需结合真实仓库做评分演练，优先使用本章案例教学包中的 `tldr`、CPython / VS Code，再回到示例包比较“为什么同是第一次贡献包，质量会明显不同”。

## 可评价证据

- 一个真实项目的《第一次贡献目标分析》。
- 一套真实贡献证据或结构完整的模拟贡献包。
- 一条真实或模拟的 review 回复，能说明如何继续在同一条公共记录中推进。
- 团队项目的《第一次贡献路径说明》。
- 团队项目的《第一次贡献候选任务清单》。

## 最小达标证据 / 提升证据

- 最小达标证据：能提交《第一次贡献目标分析》、一套结构完整的最小贡献包，以及团队项目的《第一次贡献路径说明》与候选任务清单，并解释一次 PR 需要说明哪些核心信息。
- 提升证据：能在真实或高质量模拟场景中体现清楚的边界判断、验证说明和 review 协作能力，并把这套方法迁移到团队项目的外部进入设计中。

## 常见反馈语言建议

- “你已经证明这个问题存在，但还没有证明它为什么适合现在进入。”
- “你写出了改了什么，但还没有把为什么这样改、如何验证写清楚。”
- “这条回复没有继续沿着原 PR 线程推进，公共记录链条在这里断掉了。”
- “团队项目的入口设计仍站在组内视角，外部贡献者第一次进入时还不知道先看什么。”
- “你的贡献包已经接近达标，但范围仍偏大，第一次进入更适合再收紧一个边界。”

## 基线层与扩展层建议

- 基线层：项目边界、第一次贡献入口、贡献说明、PR 组织、review 回复、持续参与。
- 扩展层：更复杂的贡献拆分、长期 Issue 跟进、权限演化、共同维护与新人引导。

## 教学结构参考

以下参考用于借鉴教学结构与支架方法，不用于照搬内容或作业设计。

- [Open Source Guides](https://opensource.guide/zh-cn/how-to-contribute/): 借鉴其把第一次参与拆成“先观察、再进入、再协作”的组织方式。
- [GitHub Docs: About forks](https://docs.github.com/en/pull-requests/collaborating-with-pull-requests/working-with-forks/about-forks): 借鉴其对 `fork` / `upstream` 路径的清楚界定，帮助学生理解外部贡献的常见入口。
- [Cornell CS5152](https://www.cs.cornell.edu/courses/cs5152/2019fa/): 借鉴其围绕真实开源项目组织团队参与与 mentor 支撑的项目式做法。
- [Producing Open Source Software](https://producingoss.com/): 借鉴其把贡献、沟通、评审与社区参与写成可复用方法的结构。
