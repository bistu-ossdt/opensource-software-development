# 第 6 章 学习指南

## 对应书稿

- 对应章节：[06-participating-in-open-source-projects.md](06-participating-in-open-source-projects.md)
- 章节主题：参与开源项目
- 使用目标：帮助学生把“想贡献”转化为“能把一次修改组织成项目可接受的贡献对象”
- 当前状态：初稿；已与稳定书稿同步一轮，可直接配合正文使用

## 如何使用本章

- 先带着“第一次贡献最先考验的其实不是编码难度”进入本章。
- 阅读时把图 6-1 当成总导航，把表 6-1 当成第一次选任务时的判断表。
- 不要只背流程名词，要顺着 `DATA_DIR` 这个贯穿案例，把“问题判断 -> 认领 -> 分支 -> commit -> PR -> checks -> review -> merge”整条链走一遍。
- 最好同时打开一个真实项目的 Issue 和一个真实 PR，边读边比照，而不是只停留在抽象概念。

## 本章学习目标

- 能解释为什么第一次参与首先是尊重项目边界，而不是急着动手。
- 能判断什么样的问题适合作为第一次贡献，什么样的问题暂时不适合。
- 能把一次修改组织成可解释、可评审、可合并的贡献对象。
- 能说明 review 为什么是协作延续，而不是简单否定。
- 能理解一次贡献如何走向持续参与与可信度积累。

## 建议阅读顺序

1. 先读第 1 节，建立“项目现在需要什么”优先于“我想改什么”的前提。
2. 再读表 6-1，把任务类型和进入前检查条件先放稳。
3. 接着读第 3 节和图 6-1，把贡献链读成一条完整对象链。
4. 再读第 4 节，重点看 review 状态、回复方式和“同一条 PR 内继续修订”的逻辑。
5. 最后读第 5 节，把一次贡献放回长期参与与角色变化的路径中。

## 关键术语与易混点

- 项目边界：不是“我会什么”，而是“项目现在接受什么、需要什么”。
- `good first issue` / `help wanted`：是入口信号，不是维护者必然接收方案的保证书。
- `fork-and-pull`：很多第一次贡献并不是直接改主仓库，而是先在自己的副本中准备修改，再向上游仓库发起 Pull Request。
- 贡献说明：不是附属文字，而是贡献对象的一部分。
- `Draft PR`：不是“半成品上传”，而是把修改方向提前放进公共记录。
- review 回复：不是礼貌补充，而是贡献真正继续发生的地方。

## 常见误区

- 第一次贡献一定要从新功能开始。
- 只要能把问题修掉，就不需要先判断项目当前节奏。
- PR 只是上传修改的入口，不需要额外说明背景和验证方式。
- 收到 `request changes` 就说明自己“做错了”。
- 第一次贡献被拒绝，就说明自己不适合继续参与。

## 如果时间有限，最少应抓住什么

- 第一次参与首先是进入项目已有节奏，而不是绕开它。
- 小而清楚、可验证的修改，往往比大而模糊的修改更适合作为第一步。
- 一次好的贡献必须能解释：解决什么问题、为什么这样改、如何验证、故意没改什么。
- AI 可以加快产出，但不会取消本章讨论的贡献路径和门禁要求。

## 本章最容易读偏的地方

- 把本章读成“如何尽快发 PR”的速成技巧。
- 把参与只理解为提交代码，不理解沟通、协商与收束。
- 把 `good first issue` 当成直接执行命令，而不再核查上下文。
- 把 review 理解成对人的否定，而不是项目上下文的回传。

## 阅读检查点

- 能否解释为什么一个真实存在的问题，仍然可能“不适合现在改”。
- 能否用表 6-1 判断一个任务是否适合作为第一次贡献。
- 能否说清 `fork`、分支、commit、PR、checks、review、merge 各自在贡献链中的作用。
- 能否根据一个 review 线程，说明贡献者下一步应该如何继续在同一条 PR 中推进。
- 能否说明第一次贡献被拒绝后，下一步为什么应先读拒绝理由，再决定换问题、等窗口，还是继续观察。

## 建议的最小实操闭环

- 选一个真实项目，找到一个看起来可能适合作为第一次贡献的问题或对象。
- 用第 1 节的方法判断它是否仍然活跃、是否仍在当前范围内。
- 用表 6-1 说明它属于哪类任务，以及为什么适合或不适合第一次进入。
- 写一段认领评论、一条 commit message、一份最小 PR 描述和一段 review 回复样稿。

## 与团队项目主线的连接

- 本章要求团队不仅维护自己的项目，也要开始用“外部贡献者如何进入”来重新审视项目。
- 团队应为自己的项目准备《第一次贡献路径说明》和候选任务清单，而不是只从内部协作视角理解仓库。
- 本章之后，团队项目至少应具备：第一次贡献候选任务、最小贡献说明、最小 review 回复约定。

## 完成本章后应留下的项目产物

- 一份《第一次贡献路径说明》。
- 一组适合作为第一次贡献的任务候选，并注明为什么适合。
- 一份最小 PR 描述模板和 review 回复示例。
- 一份《第一次贡献候选任务清单》。

## 建议的课后巩固任务

- 选一个真实项目，写出一份“我会如何做第一次贡献”的计划。
- 找出一个 `good first issue` 或 `help wanted`，用本章方法判断它今天是否仍然适合进入。
- 为团队项目设计三类第一次贡献入口：文档、测试、小型修复。
- 选一个真实 PR，分析它为什么最终被接受，或者为什么被要求返工。
