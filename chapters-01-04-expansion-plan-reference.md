# 前四章整体扩写计划参考

## 目的

本文档用于把前四章从“结构草稿 / 书稿初稿”推进到“更接近出版级技术书章节”的状态。它不是教材正文的一部分，而是书稿扩写阶段的执行参考。

若与 [00-preface.md](/home/bistu/opensource-book/00-preface.md) 冲突，以 `00-preface.md` 为准；若与 [chapter-specification-reference.md](/home/bistu/opensource-book/chapter-specification-reference.md) 冲突，以后者中的章节开发与出版级标准为准。

## 前四章的整体主线

前四章不是四个并列专题，而是一条连续主线：

1. 第 1 章回答：开源为什么形成
2. 第 2 章回答：开源如何获得制度基础
3. 第 3 章回答：开源如何被组织起来
4. 第 4 章回答：开源如何在工程上运转

对应地，四章分别承担：

- 第 1 章：历史与问题意识
- 第 2 章：概念与许可证边界
- 第 3 章：社区与治理结构
- 第 4 章：工程流程与变更控制

## 前四章的共用扩写规则

1. 扩写的目标是把前四章提升为正式技术书稿，而不是单纯加字。
2. 前四章只强化长期稳定内容，不引入变化快的工具细节、平台界面和 AI 方法。
3. 扩写应以问题、案例和判断为中心，不能靠定义堆积和空泛议论撑篇幅。
4. 正文继续保持 O’Reilly 风格技术书写法，不出现学生、教师、课堂、作业等教学词汇。
5. 前四章案例以经典开源项目为主，尤其是 GNU、Linux、Apache、Mozilla、Git，不前置现代 AI 案例。
6. 附录、实验、教学支持层的内容不能塞回正文，正文要守住技术书边界。
7. 扩写时应尽量吸收原始资料中仍有长期价值的内容，以改写融合的方式体现课程传统和历史连续性。
8. 扩写时可以吸收定位相近的参考课程和参考教材中的有价值内容，但前提是它们必须与既定章节计划直接对应，而不是因为“看起来好”就加入。
9. 吸收历史资料和外部参考时，只能吸收观点、结构、案例线索和问题组织方式，不能原文照搬，也不能做过近改写，以避免版权风险并保持全书语言统一。

## 四章统一扩写动作

每章扩写时，优先按以下顺序推进：

1. 补强开场
2. 补主体案例
3. 补概念边界与常见误区
4. 补节与节之间的递进
5. 补章末收束与下一章连接

## 分章扩写任务

### 第 1 章：开源的起源与发展

- 定位：建立历史主线与问题意识。
- 必补点：
  - 早期软件共享到软件商品化的历史转折
  - GNU / FSF 提出的核心问题
  - Linux、Mozilla、Apache 的历史角色分化
  - 1998 年后为什么必须重新理解“开源”
- 可补点：
  - Microsoft 与 GitHub 的象征意义再展开一层
  - 自由软件运动与开源运动的连续性 / 差异再补一层
- 禁区：
  - 许可证条文细节
  - GitHub 工作流
  - 现代 AI 项目和 AI-native 案例
- 优先来源：
  - 旧 `02-development-of-open-source-software.md`
  - 旧 `01-overview.md` 中少量导入性内容
  - `250622` 第 2、3 部分
- 完成标志：
  - 读者读完后，能清楚回答“开源为什么出现、为什么不能只理解成公开代码”

### 第 2 章：自由软件、开源软件与许可证

- 定位：建立制度基础与许可证判断框架。
- 必补点：
  - `Free Software` 与 `Open Source` 的关系
  - 许可证为什么是开源协作的制度基础
  - GPL、MIT、Apache、BSD 的最小区分逻辑
  - 如何做出可解释的许可证选择
- 可补点：
  - `Copyleft` 与 `permissive` 的实践影响
  - 版权、专利、商标在开源语境中的边界提醒
- 禁区：
  - 复杂法律条文展开
  - 兼容性大全
  - 企业法务手册式内容
- 优先来源：
  - 旧 `01-overview.md`
  - `250622` 第 2 部分
- 完成标志：
  - 读者能做出“为什么这个项目适合这类许可证”的基本判断

### 第 3 章：开源社区与治理

- 定位：把社区治理从附属文化话题提升为开源开发的基础组成部分。
- 必补点：
  - 社区不是代码附属物
  - 角色梯度与责任分层
  - 公开、异步、可归档沟通的意义
  - `CONTRIBUTING.md`、`CODE_OF_CONDUCT.md`、模板等治理入口
  - 社区健康与维护者可持续性
- 可补点：
  - Linux / Apache 治理路径的轻量对照
  - 基金会在治理中的作用
- 禁区：
  - GitHub 功能说明
  - 首次贡献步骤细讲
  - 大量基金会 / 社区清单
- 优先来源：
  - 旧 `03-development-models-and-characteristics.md`
  - 旧 `04-participating-in-and-organizing-open-source-projects.md` 中少量组织内容
  - 旧 `05-leveraging-open-source-software-resources.md` 中少量基金会内容
  - `250622` 第 4、5 部分
- 完成标志：
  - 读者能判断一个项目是否有清晰的社区入口、治理基础和持续性条件

### 第 4 章：开源开发的基本工程流程

- 定位：把现代开源工程理解为“变更控制系统”。
- 必补点：
  - `Issue`、Branch、Commit 的逻辑链
  - Pull Request、Review、Merge 的判断机制
  - 测试、CI、自动化检查的最低工程基线
  - 依赖、密钥、权限、发布完整性的现代基线
- 可补点：
  - Linux `patch flow` 与 Pull Request flow 的抽象对照
  - `CODEOWNERS`、rulesets 的治理含义
- 禁区：
  - 平台按钮教程
  - Git 命令大全
  - 快速变化的工具盘点
- 优先来源：
  - 旧 `03-development-models-and-characteristics.md`
  - 旧 `04-participating-in-and-organizing-open-source-projects.md` 中少量工程组织内容
  - `GitHub简介`
  - `250622` 第 4 部分
- 完成标志：
  - 读者能理解现代开源工程不是“开放提交”，而是“受控变更进入主线”
