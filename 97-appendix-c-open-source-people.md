# 附录 C 开源人物谱

这一附录从人物出发，帮助读者理解开源历史是如何被提出、推动、组织和书写出来的。这里的人物并不是按知名度简单罗列，而是按他们在开源形成中的不同作用进行组织：有的人提出问题，有的人补上基础设施，有的人推动术语与组织化传播，有的人让治理和维护经验变得可解释、可继承。

与“附录 B 开源历史关键事件时间线”相比，这里更强调人物的历史位置；与“附录 A 开源人物-事件-项目对照表”相比，这里提供更完整的简介与贡献概括。每一条都尽量聚焦稳定贡献，而不写成八卦式人物小传。

## 1. 背景传统与问题形成者

### Ken Thompson

代表项目 / 组织：UNIX、Bell Labs

Ken Thompson 的重要性，首先不在于他是否直接参与了后来的开源运动，而在于 UNIX 所代表的系统传统对开源世界产生了深远影响。UNIX 早期在研究环境中的传播方式、工具化思路与系统分层方法，为后来大量自由软件与开源项目提供了技术文化背景。很多后来的开源实践，实际上都建立在“系统可读、工具可组合、环境可迁移”这样的长期传统之上。

### Dennis Ritchie

代表项目 / 组织：UNIX、C、Bell Labs

Dennis Ritchie 与 Ken Thompson 一起塑造了后来影响极大的 Unix 传统。除了 UNIX 之外，C 语言本身也成为后来自由软件与开源系统的关键技术基础。把 Ritchie 放进开源人物谱，不是因为他提出了“open source”这个词，而是因为许多后来构成开源世界的技术对象、工程习惯和系统文化，都与他的工作密切相关。

### Bill Gates

代表项目 / 组织：Altair BASIC、早期微型机软件产业

Bill Gates 并不是开源倡导者，但他在开源历史中依然具有参照意义。1976 年的《An Open Letter to Hobbyists》明确表达了软件复制、专业劳动回报与商业化之间的关系，也让软件商品化与复制控制逻辑更公开地进入讨论。之所以在本附录中保留他，是因为开源并不是在真空中出现的，它正是在这种更强烈的软件商业化趋势中被逼成一个历史问题。

### Richard Stallman

代表项目 / 组织：GNU、Free Software Foundation（FSF）

Richard Stallman 的历史位置，在于他把软件封闭化带来的问题明确表达出来，并把“自由软件”从分散抱怨提升为系统性的工程与制度计划。GNU 项目说明，自由软件不是只共享几个程序，而是要重新建设一整套可用系统；FSF 则把这一目标进一步组织化。无论后来“open source”术语如何传播，Stallman 与 GNU / FSF 仍是理解开源历史起点时无法绕开的核心人物与组织。

## 2. 术语、许可证与公共表达推动者

### Bruce Perens

代表项目 / 组织：Debian、Open Source Initiative（OSI）

Bruce Perens 在开源历史中的重要性，不只在于个人知名度，而在于他参与了从社区经验到制度表达的关键整理工作。无论是 Debian 的早期公共协作传统，还是 OSI 对开源定义的早期推动，都体现了他在许可证边界和开源公共表达上的作用。对读者而言，他的重要性主要体现在“如何把经验写成规则”这一层。

### Eric S. Raymond

代表项目 / 组织：Open Source Initiative（OSI）、《大教堂与集市》

Eric S. Raymond 的影响，主要体现在开源传播史上。通过《大教堂与集市》，他把开放式协作模式写成了极具传播力的公共叙述，使许多人第一次把开源理解为一种不同于传统封闭开发的组织方式。无论是否完全接受他的所有判断，这部作品都在开源从社群传统走向更广泛公共讨论的过程中发挥了重要作用。

### Christine Peterson

代表项目 / 组织：Foresight Institute、开源术语形成历史

Christine Peterson 最常被记住的历史位置，是在 1998 年提出 “open source” 这一术语。这个动作的重要性，不在于它替代了此前的全部历史，而在于它为一套已经存在的实践提供了更易传播的公共语言。她在开源历史中的作用，主要属于命名与框架转换：让原有问题以新的方式进入更广泛的产业与媒体语境。

### Ian Murdock

代表项目 / 组织：Debian

Ian Murdock 的代表意义，在于 Debian 让很多人第一次看到：一个大型自由软件发行版可以在公开制度、公开协作与长期维护之间建立相对稳定的平衡。Debian 并不只是一个技术集合，它同时也是一个角色分工、制度文件和社区协作传统都非常清楚的项目。Murdock 因而常被放在“制度化社区实践”的重要人物序列中理解。

## 3. 系统、平台与治理实践塑造者

### Linus Torvalds

代表项目 / 组织：Linux、Git

Linus Torvalds 是理解开源历史时最具象征性的核心人物之一。Linux 让自由软件系统在可用层面获得关键突破，而 Git 则重塑了现代分布式协作的技术基础设施。也正因为如此，Torvalds 在开源中的贡献同时落在两个维度上：一是系统级自由软件实践的可用化，二是现代开源工程流程的基础设施化。

### Brian Behlendorf

代表项目 / 组织：Apache、Apache Software Foundation（ASF）

Brian Behlendorf 的代表意义，在于他所参与推动的 Apache 实践清楚说明了开源项目如何从补丁协作走向稳定治理。Apache 之所以经典，不只是因为软件本身重要，更因为它展示了角色、共识、基金会化和长期维护如何一起构成项目公共性。Behlendorf 因此常被视为把开源协作推向制度化社区的重要人物之一。

### Mitchell Baker

代表项目 / 组织：Mozilla、Mozilla Foundation

Mitchell Baker 的重要性，主要体现在 Mozilla 如何把开放源码、组织治理和公共传播结合起来。Mozilla 不是单纯的代码开放事件，它还关心如何让开放源码项目在长期公共性、组织结构和社区参与上站稳脚跟。Baker 在这条线上代表的是一种更现代的开源角色：既关心产品与项目，也关心公共结构与治理连续性。

### Guido van Rossum

代表项目 / 组织：Python、PEP 体系

Guido van Rossum 在开源历史中的代表性，不只是因为 Python 本身广泛流行，更因为 Python 社区长期展示了项目如何在成长过程中处理权威、提案与治理问题。从 BDFL 模式到后来的治理结构转型，Python 社区让人们更清楚地看到：成熟开源项目迟早要把个人威望转化为可继承的公共治理机制。

### Chris Wanstrath

代表项目 / 组织：GitHub

Chris Wanstrath 在这一附录中被保留，主要不是为了突出个人创业史，而是为了标记开源平台化的历史阶段。GitHub 把仓库、Issue、Pull Request、Review 与 Release 更集中地放入同一入口，显著降低了全球协作的门槛，也改变了后来一代开发者对“参与开源”这件事的直观想象。平台当然不是开源本身，但它确实深刻影响了开源实践的可见形态。

## 4. 方法、维护与可持续性书写者

### Karl Fogel

代表项目 / 组织：*Producing Open Source Software*

Karl Fogel 的独特价值，在于他把许多开源项目的治理、贡献、发布与社区运行经验写成了系统可读的方法材料。对许多后来者而言，这意味着开源不再只是依靠长期浸泡才能理解的“圈内经验”，而开始变成可以学习、比较、复用的组织方法。Fogel 的工作因此非常适合与本书后半部分实践章节一起阅读。

### Nadia Eghbal

代表项目 / 组织：*Working in Public*

Nadia Eghbal 的贡献主要在于她把“维护劳动”重新放回开源讨论的中心。她提醒人们，开源并不只由代码和热度构成，还由回应、照料、解释、版本边界与长期责任构成。对于今天理解社区健康、维护者负担和数字公共基础设施，这一视角尤其重要。

## 5. 使用方式

如果读者想回答“某个人到底为开源贡献了什么”，这一附录最合适；如果想知道“这些贡献分别发生在什么历史节点上”，则应与“附录 B 开源历史关键事件时间线”一起阅读；若希望快速总览人物、事件与项目的对应关系，则可先回到“附录 A 开源人物-事件-项目对照表”。

这份人物谱并不试图给出终极名单，而是优先提供一组长期稳定、足以支撑全书阅读的关键人物入口。后续如果全书案例系统进一步扩展，这里仍可以继续补充，但补充原则应始终保持一致：优先选择那些对开源概念、制度、治理、协作基础设施与维护方法具有长期影响的人物。
