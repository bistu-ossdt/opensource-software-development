# 第 8 章 学习指南

## 对应书稿

- 对应章节：[08-organizing-and-releasing-your-open-source-project.md](08-organizing-and-releasing-your-open-source-project.md)
- 章节主题：组织与发布你的开源项目
- 使用目标：帮助学生把“公开仓库”进一步组织成“可使用、可参与、可持续维护的开源项目”
- 当前状态：初稿；已与正文、实验材料、附录支撑和案例教学包接通，可按整章配套使用

## 如何使用本章

- 先带着“为什么公开可见的仓库还不等于真正的开源项目”进入本章。
- 阅读时优先抓住表 8-1、图 8-1、README 片段、`CHANGELOG` 条目、维护计划片段和 `SECURITY.md` 片段，它们共同构成本章的六个核心抓手。
- 把 README、LICENSE、贡献入口、版本、Release、维护计划、安全基线看成同一组公共界面对象，而不是分散的仓库文件。
- 每读完一节，都问自己：如果我是陌生使用者或贡献者，我现在能否理解并进入这个项目。
- 最好对照团队项目或一个真实开源仓库边读边检查，而不是只做概念理解。
- 阅读时要不断切换到维护者视角：不是“我能看懂吗”，而是“如果项目是我维护的，我要把哪些隐含知识写出来”。

## 本章学习目标

- 能区分“公开仓库”和“可公开参与的项目”。
- 能从维护者视角理解项目入口为何必须被显式设计。
- 能解释版本、Release、Changelog 为什么属于对外承诺。
- 能说明维护计划与治理路径为什么决定项目能否持续存在。
- 能建立小型开源项目的最小安全与供应链基线意识。
- 能把这些概念转化为具体项目对象：README、LICENSE、贡献入口、版本记录、维护计划与安全说明。
- 能对一个小型开源项目完成一次最小发布前审查。

## 建议阅读顺序

1. 先读第 1 节，建立“公开不等于可参与”的前提。
2. 接着看表 8-1 和图 8-1，把“对象分类”和“外部人进入顺序”连起来理解。
3. 再读第 2 节，从 README、LICENSE、贡献入口理解项目公共界面，并重点看 README 正反例。
4. 接着读第 3 节，把版本、Release、`CHANGELOG` 与对外承诺联系起来。
5. 再读第 4 节，理解维护计划、范围控制与治理路径为何不能等项目变大后再想。
6. 最后读第 5 节，建立小型项目的最小安全与供应链基线，并把五条最低要求当作发布前检查清单。

## 关键术语与易混点

- 公共界面（Public interface）：不是 UI，而是外部人进入项目时首先看到并依赖的说明层。
- 对外承诺（External promise）：版本、Release、Changelog 共同构成项目对使用者的稳定承诺。
- 维护计划（Maintenance plan）：不是宏大路线图，而是项目如何持续响应、发布和演进的最小节奏。
- 治理路径（Governance path）：谁负责什么，未来如何扩展角色与责任。
- 最小安全基线：不是复杂制度，而是小项目也必须具备的基本安全与来源意识。
- 发布对象（Release artifact / release object）：一次版本到底交付了哪些内容，以及这些内容如何被外部人识别和使用。
- 范围控制：维护者对“不在当前版本范围内”的请求做出清楚回应，而不是默认全部接收。

## 常见误区

- 仓库已经公开，项目就已经算开源。
- README 只要写上功能介绍就够了。
- 发布就是打版本标签。
- 维护计划和治理结构等项目做大了再说。
- 小型项目不需要谈安全和供应链。
- 只要文件都存在，项目公共界面就已经成立。
- 对范围外请求说“不”就是不欢迎协作。

## 如果时间有限，最少应抓住什么

- 开源项目的关键不只是代码公开，而是让外部人能理解、使用、参与和继续维护。
- README、LICENSE、贡献入口、版本说明与维护计划共同构成项目的公共界面。
- 版本与发布是对外承诺，不是内部记录动作。
- 小项目也应建立最小安全与供应链基线。
- 维护者的工作不是把仓库“包装漂亮”，而是降低外部进入的不确定性。

## 本章最容易读偏的地方

- 把本章读成仓库规范清单，而不是维护者视角。
- 只看到文件有无，而忽略这些对象是否真正降低了外部进入的不确定性。
- 把发布与维护看成流程末尾收尾动作，而不是项目公共性的核心部分。
- 把安全与供应链误解为大组织专属内容，而忽略小项目的最小责任。

## 阅读检查点

- 能否说明为什么公开仓库不等于可公开参与的项目。
- 能否解释 README、LICENSE、贡献说明分别在解决什么不确定性。
- 能否区分“合并修改”和“形成版本发布”的不同责任。
- 能否说明小型项目为什么也需要最小安全基线。
- 能否根据 README 片段、`CHANGELOG` 条目和维护计划片段，反推出它们分别服务哪些外部判断。
- 能否解释为什么“范围控制”也是治理的一部分。

## 建议的最小实操闭环

- 选一个团队项目或一个结构清楚的小型开源项目，先按表 8-1 做一次公共界面审查。
- 判断这个项目在哪些对象上已经说清楚，哪些仍然依赖熟人默契或内部背景。
- 先补最关键的入口对象：README、LICENSE、贡献入口。
- 再补发布层对象：版本号、一次最小 `CHANGELOG` 条目、一次最小 release note。
- 接着补连续性对象：最小维护计划和最小 `SECURITY.md`。
- 最后回头检查：如果一个陌生人今天第一次进入这个仓库，他现在能否判断“这是什么、能不能用、如何参与、遇到问题找谁、当前版本稳不稳定”。

## 与团队项目主线的连接

- 本章是团队项目从“课程内协作对象”转向“面向公共世界发布对象”的收束章节。
- 团队应在这一章之后完成项目公共界面的最小整备。
- 本章要求团队开始以维护者身份看待版本、发布、响应和后续演进。
- 团队不必一开始就建立复杂治理结构，但至少要把“入口、版本、维护、安全”四层最低要求落成公共对象。

## 完成本章后应留下的项目产物

- 一份《项目发布与维护基线说明》。
- README、LICENSE、贡献入口与版本说明的修订结果。
- 一份最小安全与供应链基线清单。
- 一份最小维护计划。
- 一次正式发布前检查记录。

## 建议的课后巩固任务

- 为团队项目写出一次正式发布前检查清单。
- 对照一个真实开源项目，分析其公共界面为什么让外部人更容易进入。
- 为团队项目写出一个最小维护计划和版本节奏说明。
- 为团队项目写一个 `CHANGELOG` 条目和一份最小 release note。
- 选一个“范围之外的请求”，尝试写一段既礼貌又清楚的维护者回应。
