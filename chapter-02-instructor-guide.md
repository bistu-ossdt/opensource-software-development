# 第 2 章 教学指南

## 对应书稿与使用方式

- 对应章节：[02-development-of-open-source-software.md](02-development-of-open-source-software.md)
- 建议用途：课程制度基础单元
- 默认按一个 `100` 分钟教学单元设计；如需拆分，可分为“概念关系”与“许可证判断”两次 `50` 分钟教学

## 教学定位

本章的目标不是把学生训练成许可证专家，而是让学生形成稳定的制度判断：开源依赖公开、明确、可解释的许可证边界。

## 本章教学目标

- 让学生区分自由软件与开源软件的不同表达重心。
- 让学生理解许可证不是装饰文件，而是协作前提。
- 让学生形成对常见许可证的最小判断框架。
- 让学生能够把许可证选择与项目目标、边界、第三方来源联系起来。

## 本章与前后章的教学边界

- 本章承接第 1 章的历史定位，把问题转成制度边界，不进入社区治理与工程流程细节。
- 本章重点是形成制度判断，不提供法律意见，也不做兼容性迷宫式展开。
- 它为第 3 章的社区与治理结构建立前提：边界先清楚，协作才可能稳定发生。

## 教学重点

- “公开可见”与“标准意义上的开源”之间有制度边界。
- 第 1 节中的连续性与差异必须同时成立。
- 表 2-1 更适合用来建立判断框架，而不是做条款背诵。
- 第 4 节必须落到真实项目选择，而不是停在抽象概念。

## 易错点与难点

- 学生容易把自由软件、开源软件、免费软件混用。
- 学生容易把许可证理解成法律课程内容，从而抽离工程场景。
- 学生容易以“常用程度”代替“项目边界判断”。
- 学生容易忽略项目里的图片、模板、数据和第三方代码来源。

## 课堂上不应展开过深的内容

- 不展开过细的许可证兼容性链条与法律争议。
- 不把课堂重心放到个别条款背诵上。
- 不提前展开治理结构或 GitHub 工作流细节。

## 推荐课堂组织

- `10` 分钟：用“公开仓库但没有 `LICENSE`”的场景开场。
- `25` 分钟：讲自由软件与开源软件的连续性与差异。
- `25` 分钟：讲许可证的制度作用，并结合表 2-1 解释常见选择。
- `20` 分钟：做一次项目化选择讨论，例如“这个团队项目为什么不应等到发布前再补许可证”。
- `20` 分钟：把本章收束到“制度边界是后续治理与流程的前提”。

## 推荐课堂活动

- 活动 1：给学生若干仓库场景卡片，判断“公开可见”“有源码下载”“有许可证”“构成标准意义上的开源”分别是否成立。
- 活动 2：围绕 MIT、Apache-2.0、GPL 做三角比较，要求学生只说明“各自优先保护什么边界”，不做条款背诵。
- 活动 3：让团队为自己的项目做一次最小来源清点，区分代码、图片、数据、模板和第三方依赖。
- 活动 4：给出“仓库公开但无许可证”的反例，要求学生写出为什么这不是“默认开放”。

## 推荐讨论问题

- 为什么“代码能看到”并不足以构成开源？
- 为什么无许可证状态比许多学生想象中更严格？
- 宽松式许可证与 Copyleft 许可证分别在保护什么边界？
- 一个项目的许可证选择，为什么必须和其目标、协作方式、第三方来源一起讨论？

## 案例使用建议

- 可以使用“公开仓库但无许可证”的反例。
- 适合用 MIT、Apache-2.0、GPL 做三角比较，不必扩展到过多许可证。
- 若使用真实仓库示例，应优先使用许可证边界清楚、项目目标明确的项目。

## 与团队项目主线的阶段映射

- 本章是团队项目第一次真正进入制度设计的节点。
- 本章结束后，团队应能说明当前项目为什么适合采用某种许可证，以及项目对象的来源状态如何。
- 本章对应的阶段产物是《许可证与来源清点说明》。

## 评分关注点建议

- 核心看学生是否把许可证放回项目边界，而不是背诵定义或只凭流行程度选择。
- 对“无许可证”场景的判断应明确指出：公开可见不等于默认授权。
- 团队项目说明应至少覆盖项目目标、协作方式、第三方来源和对象类型，而不是只写一个许可证名称。

## 示例包与课堂校准

- 建议先展示一份“只写许可证名”的弱样例，再展示一份“把项目目标、来源和边界一起说明”的强样例。
- 校准表 2-1 时，重点不是问学生记住多少条款，而是看能否稳定解释“这个项目为什么更适合这类边界”。
- 如需协同批改，建议先用一组真实或模拟仓库对象统一判断“无许可证”“宽松式许可证”“Copyleft 许可证”三类基本场景。

## 可评价证据

- 一页“公开仓库是否构成开源项目”的判断说明。
- 一页常见许可证对比说明。
- 团队项目的许可证选择备忘录。

## 最小达标证据 / 提升证据

- 最小达标证据：能提交《许可证与来源清点说明》初稿，并解释“无许可证”为什么不等于默认开放。
- 提升证据：能比较常见许可证的边界差异，并把选择理由与项目目标、第三方来源和开放边界清楚联系起来。

## 常见反馈语言建议

- “你已经写出了许可证名称，但还没有说明它和项目边界之间的关系。”
- “这份判断把公开仓库直接等同于开源项目了，缺少对授权边界的说明。”
- “你比较了几个许可证，不过比较维度还停留在流行度，没有回到项目目标和协作方式。”
- “来源清点已经开始做了，但图片、模板、数据和第三方代码还没有分开看。”

## 基线层与扩展层建议

- 基线层：自由软件 / 开源软件关系、许可证作用、MIT / Apache-2.0 / GPL 的最小区分、无许可证风险。
- 扩展层：MPL / LGPL / AGPL、更复杂的兼容性问题、专利与 `NOTICE`、项目对象类型差异。

## 教学结构参考

以下参考用于借鉴教学结构与支架方法，不用于照搬内容或作业设计。

- [Stanford CS101](https://web.stanford.edu/class/cs101/): 借鉴其“主讲概念 + 阅读材料 + 书面练习”的结构，把概念区分与制度判断拆开处理。
- [Stanford CS101 Big Ideas](https://web.stanford.edu/class/cs101/big-ideas.html): 借鉴其“先抓核心对比，再进入细节”的组织方式，本章宜先建立大框架，再补许可证差异。
- [CS50 Syllabus](https://cs50.harvard.edu/college/2026/spring/syllabus/): 借鉴其讲授后配套综合问题与小组支持的方式，本章适合用短讨论或判断题帮助学生做概念综合。
