# 第 8 章 案例教学包

## 对应章节与用途

- 对应书稿：[08-organizing-and-releasing-your-open-source-project.md](08-organizing-and-releasing-your-open-source-project.md)
- 配套学习指南：[chapter-08-study-guide.md](chapter-08-study-guide.md)
- 配套教学指南：[chapter-08-instructor-guide.md](chapter-08-instructor-guide.md)
- 本案例包用于帮助学生把“组织与发布项目”理解为一组真实的维护者对象：公共界面、版本承诺、维护计划与安全边界，而不是把课程最后一章讲成形式化收尾。

## 主案例与辅助案例

| 案例 | 本章主要说明什么 | 推荐观察对象 |
| --- | --- | --- |
| `tldr-pages/tldr` | 小型项目如何把公共界面做得清楚、轻量且可进入 | `README.md`、`LICENSE.md`、`CONTRIBUTING.md`、Issue / PR 入口、版本记录 |
| Linux | 长期项目如何通过维护链、发布线和文档把项目持续交给公共世界 | `MAINTAINERS`、`Documentation/process/`、稳定版本线、长期维护角色 |
| CPython | 成熟项目如何把版本、发布节奏、支持版本和安全处理组织成长期承诺 | 版本生命周期、发布说明、开发者文档、安全相关说明 |
| OpenClaw | 当对象本身含 AI 部分时，发布说明为什么必须把公开边界说清楚 | `README`、安装与运行说明、代码 / 参数 / 数据信息边界、已知限制与项目状态 |

## 推荐课堂使用方式

- 先用 `tldr-pages/tldr` 讲“小项目也可以有清楚公共界面”，因为它最适合让学生快速看到 README、LICENSE、贡献入口如何共同工作。
- 再用 Linux 讲“可持续维护”不是抽象口号，而是维护链、版本线和公开文档长期积累的结果。
- 然后用 CPython 讲“版本号、支持版本和安全处理为什么属于长期对外承诺”，帮助学生理解第 3-5 节并不是仓库装饰。
- 最后用 OpenClaw 收住“AI 项目发布边界”这条支线，提醒学生：只要对象本身含 AI 部分，就必须把开放到了什么程度写清楚。
- 课堂最后回到团队项目，让学生用这四类案例反向审视自己的仓库：入口是否清楚、版本是否可解释、维护是否可持续、AI 边界是否被写明。

## 与书稿贯穿对象的配合

- 第 1 节的“公开仓库不等于可公开参与项目”最适合用 `tldr-pages/tldr` 与一个入口薄弱仓库做对比。
- 表 8-1 与图 8-1 适合和 `tldr-pages/tldr` 一起使用，帮助学生把公共界面对象与进入顺序对应起来。
- 第 3 节的版本与发布承诺，适合配合 CPython 或 Linux 的长期发布线来理解“版本不是内部编号”。
- 第 4 节的维护计划与治理路径，适合用 Linux 的 `MAINTAINERS` 和稳定版本线说明“维护者连续性”。
- 第 5 节的安全与供应链基线，适合用 CPython、Linux 或课程团队项目自己的 `SECURITY.md` 草稿来讲“小而真实”的安全责任。
- OpenClaw 主要承担本章的 AI 支线：当项目对象本身含 AI 部分时，发布说明必须把公开边界讲清，而不能只写“已经开放”。

## 推荐讨论问题

- 一个小项目为什么也必须把 README、LICENSE、贡献入口和版本说明放在同一层看待？
- 为什么 Linux 这样的长期项目会把维护角色、稳定分支和文档长期外显出来？
- 版本号什么时候只是标签，什么时候才开始成为对外承诺？
- 为什么“项目还在快速变化”也应被直接写出来，而不是留给外部人自行猜测？
- 对于含 AI 部分的项目，为什么发布说明比传统软件更需要把边界写清？

## 与团队项目主线的连接

- 团队可借 `tldr-pages/tldr` 反推自己的 README 是否已经回答关键问题。
- 团队可借 Linux 理解“没有长期维护链，公开发布也很容易迅速失速”。
- 团队可借 CPython 理解“支持版本”和“安全说明”为什么不是大项目专属内容。
- 若团队项目本身含 AI 部分，可借 OpenClaw 检查自己的项目是否清楚说明了代码、参数、数据信息与运行边界。

## 使用边界

- 不要把案例包讲成“哪个项目更伟大”的历史介绍，应始终回到本章的维护者对象。
- 不要因为 Linux 或 CPython 太大，就把它们变成需要全面读懂的主案例；这里借的是结构，不是要求学生掌握全部内部实现。
- 不要让 OpenClaw 喧宾夺主；它在本章只承担 AI 发布边界说明，不替代公共界面与维护计划主线。
- 不要把课堂变成“照着优秀项目抄文件”，而应引导学生先回答自己项目当前真实状态是什么。
