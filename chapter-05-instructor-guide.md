# 第 5 章 教学指南

## 对应书稿与使用方式

- 对应章节：[05-leveraging-open-source-software-resources.md](05-leveraging-open-source-software-resources.md)
- 建议用途：前半部稳定核心进入后半部参与实践的过渡单元
- 默认按一个 `100` 分钟教学单元设计；如需拆分，可分为“项目入口判断”与“健康度 / 风险 / 行动选择”两次 `50` 分钟教学
- 当前状态：较完整初稿；已与正文、实验材料、附录支撑和案例教学包同步，可直接组织一轮完整教学

## 教学定位

本章不是“教学生逛 GitHub 页面”，而是帮助学生建立面对陌生开源项目时的整体阅读方法。它是从前四章稳定知识转向真实项目判断能力的桥梁。

## 本章教学目标

- 让学生理解项目阅读首先是判断项目为何存在、当前状态如何，而不是立即进入代码。
- 让学生理解入口对象、治理信号和工程对象如何共同构成整体证据链。
- 让学生能够区分热度、健康度、可参与性与风险。
- 让学生把项目阅读结果落实为下一步行动选择。

## 本章与前后章的教学边界

- 本章不讲具体平台操作技巧，也不展开复杂代码架构分析。
- 本章重点是“如何形成项目级判断”，不是“如何立即提交贡献”。
- 如果课堂上已经出现“怎么认领、怎么写 commit message 或 PR 描述”的问题，应明确记下并推迟到第 6 章集中处理。
- 它为第 6 章的第一次真实参与做准备，是后半部实践章节的入口。

## 教学重点

- 第 1 节必须讲透“项目阅读首先是评估活动”。
- 图 5-1 要作为第一轮阅读顺序的课堂骨架，而不是只当插图出现。
- 第 2 节中的入口层对象要讲成“第一层证据”，而不是文件列表。
- 正文里的 `tldr-pages/tldr` 行走式阅读演示要真正走一遍，不能只在口头上提到。
- 图 5-2 与表 5-1 要反复用于“零散对象 -> 项目判断 -> 行动选择”的收束。

## 易错点与难点

- 学生容易先跳进代码细节，而忽略项目定位。
- 学生容易把 star 数误当成项目质量和可参与性的直接证据。
- 学生容易只检查对象有无，而不判断这些对象之间是否一致。
- 学生容易把项目阅读理解成“看懂代码越多越好”，而忽略决策价值。
- 学生容易把 AI 项目中的“开放权重”误读成完整的开源人工智能对象。

## 课堂上不应展开过深的内容

- 不应花太多时间在页面导航式演示上。
- 不应把本章变成代码架构精读课。
- 不应提前进入第 6 章的 PR 提交流程和 review 回复细节。
- 不应要求学生在本章提交认领评论、commit message 或 PR 描述样稿。

## 推荐课堂组织

- `10` 分钟：从“为什么很多人看过仓库却仍不知道能不能参与”开场。
- `20` 分钟：讲第 1 节和图 5-1，建立“项目定位 -> 入口对象”主线。
- `20` 分钟：用 `tldr-pages/tldr` 演示第 2 节中的第一轮阅读路径。
- `20` 分钟：讲第 3 节、图 5-2 和表 5-1，示范如何把分散对象重新连起来。
- `15` 分钟：讲第 4 节，区分热度、健康度、可参与性与风险。
- `15` 分钟：讲第 5 节，让学生把判断压缩成行动选择。
- `10` 分钟：让学生把同样的方法映射到团队项目。

## 推荐课堂活动

- 活动 1：用 `tldr-pages/tldr` 做一次完整的第一轮阅读，要求学生按图 5-1 填一份最小阅读记录。
- 活动 2：对比 `tldr-pages/tldr` 与 VS Code，判断两者的公共入口、治理对象和工程对象为什么分布不同。
- 活动 3：用 Linux 与 OpenClaw 做“证据分布位置”对照，说明长期基础设施项目与现代 AI 原生项目的入口不在同一层。
- 活动 4：让团队从“陌生读者”视角重读自己的项目，写出《公共入口缺口清单》，并只记录可能进入第 6 章的候选对象，不在本章展开贡献演练。

## 推荐讨论问题

- 一个仓库“看起来很热闹”，为什么仍可能不适合参与？
- README、Release、Issue、PR 分别在说明项目的什么状态？
- 为什么“缺某个文件”和“项目不可进入”之间不能简单画等号？
- 项目阅读的结果为什么必须落到行动选择上？
- 为什么不同类型项目公开协作证据的位置并不相同？

## 案例使用建议

- 课堂顺序固定为：`tldr-pages/tldr` -> VS Code -> Linux -> OpenClaw。
- `tldr-pages/tldr` 用于讲“低成本第一轮阅读路径”；VS Code 用于讲 GitHub-native 成熟项目的公共对象；Linux 用于讲长期维护网络；OpenClaw 用于讲现代 AI 项目的公共入口与边界判断。
- 案例必须和书稿正文已有对象一致，不再额外引入无关仓库。
- 不建议把课堂重点放在具体平台界面点击步骤上。
- 本章案例只用于阅读判断与行动选择，不用于认领评论、PR 结构和 review 演练。

## 与团队项目主线的阶段映射

- 本章要求团队第一次从“陌生读者”视角审视自己的项目。
- 本章结束后，团队至少应知道当前项目是否已经具备最小公共界面。
- 本章对应的阶段产物是《陌生项目阅读记录》《项目阅读摘要》和《公共入口缺口清单》。

## 评分尺使用建议

- 本章评分以 [chapter-05-lab-project-materials.md](chapter-05-lab-project-materials.md) 中的实验级评分尺为准，按 `5` 个维度、每项 `20` 分组织。
- 建议评分顺序固定为：先看《陌生项目阅读记录》，再看《项目阅读摘要》与行动选择，最后看团队项目的《公共入口缺口清单》。
- 不要把“项目名气大”或“浏览得很久”当作直接加分项；核心看证据组织、判断质量与行动选择是否成立。
- 若使用课程提供的仓库快照而不是真实在线仓库，不应直接扣分；应根据对象是否真实、判断链是否完整来评估。
- 若缺少《陌生项目阅读记录》《项目阅读摘要》与《公共入口缺口清单》三项中的任一项，通常不判为“完整达标”。

## 示例包与课堂校准

- 附录支撑中的示例包可按“常见问题样例 -> 达标样例 -> 高质量样例”的顺序使用，帮助学生先挑错，再校准最低通过线，最后理解优秀标准。
- 更稳妥的做法是先围绕 `tldr-pages/tldr` 做一次班级统一校准，再让学生迁移到自己选择的真实项目。
- 如果课堂时间有限，至少要让学生比较一组“好阅读摘要 / 坏阅读摘要”和一组“好行动选择 / 坏行动选择”，这样评分标准才不会停留在抽象口号。
- 如需做助教或多教师协同批改，建议所有评分人员先共同阅读 [chapter-05-appendix-support.md](chapter-05-appendix-support.md) 中的示例包，再开始正式判分。

## 可评价证据

- 一页《陌生项目阅读记录》。
- 一页《项目阅读摘要》。
- 一个真实项目的健康度 / 可参与性 / 风险三项判断。
- 团队项目的《公共入口缺口清单》。

## 最小达标证据 / 提升证据

- 最小达标证据：能完成一页《陌生项目阅读记录》和《项目阅读摘要》，并说明一个项目是否适合继续学习、尝试贡献、复用、先观察或暂缓。
- 提升证据：能比较两个项目的入口层、治理层和工程层差异，并把判断迁移到团队项目改进上。

## 常见反馈语言建议

- “你已经记录了仓库里有什么，但还没有把这些对象压缩成一个项目级判断。”
- “这份摘要说明了项目功能，但还没有说明项目当前是否值得继续投入。”
- “你的行动选择已经写出来了，但理由仍停留在印象层，还没有回到仓库证据。”
- “你指出了团队项目的缺口，但还没有说明哪个缺口最先影响外部人进入。”
- “这份阅读记录覆盖面够广，但还需要再收束：哪些对象是真正决定你判断的关键证据？”

## 基线层与扩展层建议

- 基线层：项目定位、入口对象、整体判断、健康度 / 可参与性 / 风险、行动选择。
- 扩展层：更深入的架构分层阅读、模块边界分析、复杂治理信号、依赖风险与 AI 项目边界讨论。

## 教学结构参考

以下参考用于借鉴教学结构与支架方法，不用于照搬内容或作业设计。

- [Open Source Guides](https://opensource.guide/zh-cn/how-to-contribute/): 借鉴其把“参与前先观察项目状态”讲成进入开源的前置动作。
- [GitHub Docs: Exploring projects on GitHub](https://docs.github.com/en/repositories/working-with-files/using-files/exploring-projects-on-github): 借鉴其把仓库入口对象组织成可扫描阅读路径的方式。
- [GitHub Docs: About community profiles for public repositories](https://docs.github.com/en/repositories/managing-your-repositorys-settings-and-features/customizing-your-repository/about-community-profiles-for-public-repositories): 借鉴其把 README、许可证、贡献说明、安全策略等对象视为公共入口信号的方式。
- [Cornell CS5152](https://www.cs.cornell.edu/courses/cs5152/2019fa/): 借鉴其围绕真实项目建立阅读、判断与贡献路径的组织方式。
- [CS50 Syllabus](https://cs50.harvard.edu/college/2026/spring/syllabus/): 借鉴其“主讲 + 任务 + 综合反思”的课堂支架结构。
