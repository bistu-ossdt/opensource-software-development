# 00 课程整体设计原则与编写约束

> 当前状态：本文件在课程设计与教材编写阶段充当整体课程设计原则与编写约束文档，覆盖教材、附录、实验材料和配套案例。教材完稿后，可将其中面向学生的部分改写为教材“前言”。

## 课程与教材定位

本课程面向大学本科生，“开源软件开发技术”教材是本课程的核心载体之一，但不是课程的全部。本课程不是泛化的软件开发课程，也不是单纯的 AI 编程工具课程，而是通过教材、附录、实验材料和配套案例，帮助学生理解开源、进入开源、开始参与开源的软件开发课程。

本课程希望帮助学生完成三个过渡：

- 从“听说过开源”到“理解开源文化、历史与社区逻辑”
- 从“会写程序”到“掌握开源开发的基本技术与协作流程”
- 从“课程学习者”到“能够开始参与和开发开源软件的贡献者”

## 课程历史的继承与重构

本课程的新开发，不应被理解为对既有课程历史的替代，而应被理解为在十余年课程实践基础上的继承、重构与现代化升级。

这意味着：

- 既有课程中的有效经验、核心传统、代表性实践和优秀成果，应被视为课程资产，而不是历史包袱
- 旧讲义、旧实践、历年项目仓库和长期形成的教学方法，应进入当前课程设计的分析、迁移和重写过程
- 新课程要保留这门课程长期形成的身份连续性，而不是完全抛弃原有积累，另起炉灶
- 对历史材料的处理原则不是简单保留或简单否定，而是判断哪些内容值得继承，哪些需要重构，哪些需要按现代开源开发实践升级

因此，本课程既是一个“新开始”，也是一门具有十余年教学连续性的课程在新时代条件下的系统升级。

## 课程学习成果

本课程的设计应落实为可观察、可评价的学习成果。完成课程后，学生原则上应能够：

- 解释自由软件（Free Software）与开源软件（Open Source Software）的关系与差异
- 理解开源运动的历史脉络、关键人物、代表性事件与代表性项目
- 识别常见开源许可证（如 GPL、MIT、Apache、BSD）的基本特点与适用场景
- 阅读一个真实开源项目的代码仓库、README、Issue、Pull Request 与贡献规范
- 基于 Git 与 GitHub 完成基本的开源协作流程，如 Fork、Branch、Commit、Pull Request、Review 与 Merge
- 理解现代开源仓库中的基本治理机制，如 `CONTRIBUTING.md`、`CODE_OF_CONDUCT.md`、`SECURITY.md`、Issue / Pull Request 模板、`CODEOWNERS`、分支保护与规则集（rulesets）
- 为一个开源项目补充文档、测试、小型功能或缺陷修复
- 理解测试、CI、代码评审、发布与维护在开源开发中的作用
- 理解依赖审查（dependency review）、密钥保护（secret scanning / push protection）、软件物料清单（SBOM）、构建来源证明（provenance / attestations）等现代开源工程机制的基本意义
- 在开源流程和规范约束下合理使用 AI 辅助开发
- 能够从参与者视角和维护者视角理解开源项目的协作与持续发展

## 学生前置基础

本课程默认主要面向大学二年级及以上学生。为保证教学节奏和实践效果，默认学生应具备以下基础：

- 至少掌握一门编程语言的基础语法与基本编程能力
- 具备基本命令行使用能力
- 具备基本程序阅读能力

课程及教材不以“零编程基础”作为前提来设计内容。

## 历史学生数据的使用边界

课程设计可以参考历年学生项目、课程仓库和学习表现，但这类数据的作用应被严格限定。

历史学生数据的价值在于：

- 为课程设计提供现实证据，帮助判断学生通常具备哪些起点能力
- 帮助识别哪些能力已经有实践基础，哪些能力长期是薄弱环节
- 为课程节奏、任务难度、支架强度和实验组织方式提供校准依据

但历史学生数据 **不应** 被用来直接决定课程目标上限，也不应使课程设计被某一届学生的即时水平所绑架。

本课程的目标仍应主要由以下因素共同决定：

- 本科阶段开源软件开发课程的培养定位
- 开源文化、社区和工程实践的长期稳定要求
- 现代开源开发技术与治理基线
- 学生从“进入开源”到“开始参与开源”所需的真实能力结构

因此，历史学生数据在本课程中的角色，是增加设计维度、增强现实依据，而不是替代课程目标本身。

## 适用对象与内容弹性

本课程虽然统一面向大学本科生，但在适用对象上应有明确定位：课程整体应主要面向具备正常计算机专业基础的一般本科院校，同时保留一部分可向上扩展的内容，以满足高水平院校或能力较强学生的学习需要。

这意味着课程内容应具有“基线层 + 扩展层”的弹性结构：

- 大部分内容应作为通用基线，能够被一般本科院校稳定开设和实施
- 一部分内容应作为扩展提升，支持高水平院校或高能力学生进行更深入的项目实践、治理分析、工程化训练与 AI 辅助开发探索
- 课程设计不以“一般以下”的基础条件作为主要适配对象，不因追求更低门槛而削弱课程的学科定位与能力要求

这种弹性不应通过降低主线标准来实现，而应通过内容分层、案例分层、实验分层和评价分层来实现。

具体来说：

- 正文应以多数本科院校可稳定讲授的长期内容为主
- 附录与案例可提供更丰富、更前沿的扩展材料
- 实验可设置“必做基础项 + 选做提升项”，以适应不同学校与不同学生团队的能力差异

课程的主目标仍然统一，即帮助本科生理解开源、进入开源、并开始以现代开源方式参与和开发开源软件。

## 国际对齐与双语一致性

本课程未来将建设中英文双语版本，因此从现在开始，课程内容结构、材料类型、章节规范和术语体系都应尽量与国际上主流大学的计算机课程组织方式保持对齐，而不应形成一套只适用于本地语境、难以转化为英文版的特殊结构。

这里的“国际对齐”主要指：

- 课程结构应尽量对齐国际主流大学课程常见的组织方式，如明确的学习目标、核心概念、案例、实践接口、延伸阅读与可评价成果
- 内容安排应尽量对齐国际开源教育、软件工程教育和技术书写作中的主流表达习惯
- 材料类型命名应尽量使用国际可对照的结构，如技术书稿、Student Study Guide、Instructor Guide、附录、实验材料、案例库等
- 参考文献、案例来源和术语解释应优先连接国际上可验证、可追溯、可复用的来源体系

这里的“双语一致性”主要指：

- 中文版与英文版应共享同一套课程结构骨架，而不是分别独立设计
- 章节标题、节标题、术语表、案例类型、实验类型和交付物名称都应尽量做到中英文可一一对应
- 关键术语应优先以国际通行英文原词为基准，再给出准确、稳定的中文表达
- 若中文表述与国际通行术语存在偏差，应优先修正中文，而不是迁就本地习惯造成后续英文版失真

因此，本课程中的中文写作从一开始就不应被视为一个纯中文项目，而应被视为未来中英文双语课程体系中的一个版本。结构、术语和材料分层都应为双语扩展预留一致性。

## 内容取向

为了使课程体系和教材内容能够经受时间考验，本课程前半部分优先讲述长期稳定、不易过时的内容，包括：

- 开源文化、历史与价值
- 开源运动中的重要事件、重要人物与代表性项目
- 自由软件与开源软件的基本概念
- 许可证、版权、专利、商标
- 开源社区的协作逻辑与治理方式
- 版本管理、测试、持续集成、发布维护等基本工程机制

这些内容构成课程与教材的稳定内核，应尽量少依赖具体平台界面、短期热点或快速变化的工具细节。

本课程还应明确把“社区治理”从开源文化的附属话题提升为开源软件开发技术的基础组成部分。因为在开源场景中，角色分工、决策机制、贡献路径、行为规范、透明度和社区健康并不是外围问题，而是开放协作得以长期运行的组织基础和工程前提。

## AI 内容的安排

AI 已经深刻影响软件开发，但本课程关注的不是“如何用 AI 开发任何软件”，而是“如何在开源软件开发中使用 AI”。因此，AI 相关内容放在课程后半部分，作为偏实践、更新较快的章节处理，必要时可以扩展为两到三章。

AI 部分的重点应包括：

- 如何用 AI 理解开源项目的代码、文档、Issue 和 Pull Request
- 如何用 AI 辅助补文档、补测试、修复缺陷和实现小功能
- 如何通过 GitHub、测试、CI、Review 等流程管理 AI 生成内容
- 如何在开源社区的规范、许可证和工程流程下使用 AI

## 软件工程内容的边界

本课程可以介绍软件工程，但只讲与开源软件开发直接相关、且本科生必须掌握的部分，例如：

- Git 与 GitHub 为代表的版本管理与协作流程
- Issue、Branch、Commit、Pull Request、Code Review、Merge、Release
- 测试、CI、缺陷跟踪、文档和维护

本课程及其教材不追求覆盖完整的软件工程知识体系，而强调这些技术在开源软件开发中的实际作用。

## 现代开源开发技术基线

本课程应明确体现这样一个判断：现代开源开发技术已经不再等同于“公开代码仓库 + 接收 Pull Request”的旧式协作模式，而是逐渐演变为一个受治理、可验证、重视安全与供应链完整性的工程系统。

因此，课程中对“开源开发技术”的理解应至少覆盖以下现代基线：

- 仓库治理文件与社区健康机制
- 基于 Pull Request 的变更控制与评审门禁
- 测试、持续集成与自动化质量检查
- 依赖、漏洞、密钥与权限等安全机制
- 发布、软件物料清单（SBOM）与构建来源证明等供应链实践

这些内容在课程中的展开深度可以根据对象和学时调整，但不应被视为可有可无的附加主题。

## 非目标与范围边界

为避免课程定位漂移，本课程明确不以以下内容作为主体目标：

- 不写成覆盖全面的软件工程教材
- 不写成以工具盘点为主的 AI 编程工具手册
- 不写成面向企业内部闭源研发流程的开发教材
- 不把短期热点产品界面、功能清单和厂商策略作为正文主干

教材、附录和实验材料可以适度借用相关内容，但前提是它们直接服务于本科生理解和参与开源软件开发。

## 本科生导向

本课程围绕本科生第一次进入开源世界的真实学习路径组织内容。教学重点是帮助学生完成以下关键起步：

- 第一次理解开源发展的历史脉络和关键人物的作用
- 第一次读懂一个开源项目
- 第一次提交文档、测试或小型代码修改
- 第一次发起 Pull Request 并参与评审
- 第一次理解许可证和贡献规范
- 第一次组织并发布一个小型开源项目

内容安排应循序渐进，以可上手、可实践、可复用为原则，并尽量降低学生第一次进入真实开源项目时的认知负担。

## 认知与学习维度

本课程不仅要考虑“教什么”，还要考虑学生如何真正学会。对本科生而言，开源软件开发的难点往往不只是技术本身，还包括陌生仓库、陌生术语、英文文档、协作规范、公开反馈与真实社区环境带来的认知压力。

因此，课程与教材设计应遵循以下认知与学习原则：

- 从可理解的小范围任务切入，逐步过渡到更完整的项目参与
- 先帮助学生学会阅读、观察、理解，再要求其提交和维护
- 通过真实但可控的案例降低进入开源世界的心理门槛
- 尽量避免在早期章节同时引入过多新概念、新工具和新流程
- 通过术语解释、案例支架、流程示意和阶段性任务帮助学生建立整体认知地图

认知与学习维度的目标，不是降低课程标准，而是让学生能够从“看不懂、不敢做”逐步过渡到“能理解、能参与、能持续贡献”。

## 现代计算机工程师能力融合

本课程虽然以开源软件开发为中心，但也希望通过开源场景，融合培养现代计算机工程师应具备的核心能力。这里的“能力融合”不是把教材扩展成泛化的软件工程大全，而是将这些能力落实在开源软件开发的真实流程中。

应重点培养的能力包括：

- 阅读代码、文档与技术资料的能力
- 版本管理、协作开发与代码评审能力
- 测试、调试、自动化与质量保障能力
- 使用命令行、开发工具和工程环境的能力
- 技术沟通、文档表达与社区协作能力
- 对许可证、合规、安全和学术诚信的基本判断能力
- 在现代工具环境下合理使用 AI 辅助开发的能力

这些能力不应脱离开源实践单独讲授，而应尽量通过仓库、社区、文档、Issue、Pull Request、测试、CI 和项目维护等具体场景自然体现。

## 实践衔接方式

本课程应具有较强实践性，但教材本身以内容组织和能力支撑为主，具体实验、作业和课堂任务可以作为独立教学材料安排。教材内容应当能够自然呼应这些实践活动。

建议实践采用由浅入深的三级衔接方式：

- 第一层：阅读、观察和分析真实开源项目，理解文化、历史、人物、许可证和社区逻辑
- 第二层：在课程仓库或教学环境中进行开源协作流程演练，如文档、Issue、Branch、Pull Request、测试和 CI
- 第三层：在真实或近真实的开源场景中完成小型贡献、项目发布或维护实践

每层实践应能对应可评价的学习成果，使教师能够判断学生是否达到了该层次的预期能力。

## 团队开源项目主线

本课程应支持学生以团队形式完成一个开源项目的规划、协作开发、测试、发布和维护，因此团队开源项目不应只是课程末尾附加的综合作业，而应成为贯穿课程的重要实践主线。

这意味着课程内容除讲授开源文化、社区、治理和开发流程外，还应显式覆盖以下项目化能力：

- 项目选题与问题定义
- 范围控制与最小可发布版本（MVP）
- 团队角色分工与协作机制
- 任务拆分、迭代规划与版本安排
- 面向使用者的文档、发布与交付
- 项目复盘、维护与后续演进

课程中的章节学习、附录材料、实验安排和课程考核都应尽量与这条团队项目主线形成对应关系，使学生能够在学习过程中逐步完成一个真实或近真实的开源项目。

## 课程材料体系

本课程应形成相互配合的材料体系，而不是只完成一本教材。基于当前目标，课程材料体系应明确升级为以下几类正式产物：

- `Technical Book Manuscript`：技术书稿，作为可独立成立、可出版的主书稿，承载长期稳定的核心知识、方法与案例解释；其默认对象是技术读者，写法应接近 O'Reilly 风格的技术书，而不是课堂讲义；其难度基线按具备本课程先修条件的读者设定，并通过“基线层 + 扩展层”保持内容弹性
- `Student Study Guide`：学生学习指南，承载如何使用书稿学习、如何把书稿内容连接到课程项目与实验、常见误区与学习建议
- `Instructor Guide`：教师教学指南，承载授课建议、讨论重点、案例选择、实验映射、评价证据与教学组织建议
- `Course Syllabus / Course Guide`：课程说明层，承载课程简介、周次安排、考核结构、项目里程碑、支持方式与政策信息
- `Appendices`：附录，承载更新较快的工具、平台、术语、命令、模板、清单与补充案例
- `Lab / Project Materials`：实验与项目材料，承载实验说明、项目要求、课堂任务、提交规范、评分标准与项目阶段推进
- `Case Library`：案例库或项目库，承载真实开源项目、Issue、Pull Request、许可证、社区治理与现代仓库治理案例

这些材料应围绕同一份课程设计原则协同建设，并在后续迭代中保持一致。技术书稿是主文本，其他层是围绕主文本组织学习、教学、课程运行与实践支撑的配套层，而不是彼此平行独立发展的新正文。

在材料定位上，还应明确区分两类对象：

- 课程层材料面向教学场景，服务课程组织、学习支架、课堂实施、实验安排、项目推进与评价
- 书稿层材料面向技术读者，服务开源软件开发知识、方法、案例与实践判断的独立阅读

这里还应进一步区分三个不同维度，而不应混为一谈：

- 书稿层的“对象”是广义技术读者，而不是仅限于课堂中的学生
- 书稿层的“难度基线”按具备大学计算机专业基础、满足本课程先修条件的读者设定
- 书稿层的“内容弹性”通过“基线层 + 扩展层”实现：正文主干服务一般本科院校可稳定掌握的内容，扩展内容则支持更强读者、高水平院校和更深入的实践场景

这意味着：

- 课程层材料可以明确使用“学生”“教师”“教学”“课堂”“作业”“实验”等表述
- 书稿层材料不应显式把读者写成学生或教师，而应默认面向希望系统了解开源软件开发技术的技术读者
- 书稿层材料虽然服务本科课程，但写法上应尽量保持技术书的自然叙述、问题驱动与案例驱动风格，而不是转化为课堂管理语言
- 书稿层材料不应被误写成“只适用于本科课堂”的教材，也不应被写成默认面向资深专家的高门槛手册

## 整体课程设计与分层承载规则

本课程应先进行整体设计，再按不同材料层次分工承载内容，以同时保证课程整体性、技术书稿质量、教学可实施性与双语扩展能力。

### 课程级材料

以下内容属于课程级材料，不应混入技术书稿正文：

- `Course Syllabus / Course Guide`：课程目标、模块安排、周次、考核结构、项目里程碑、迟交与学术诚信政策、支持方式
- `Instructor Guide` 中的课程组织信息：授课节奏、课堂讨论组织、案例使用策略、评分与反馈建议
- 跨章节的 `Lab / Project Materials`：实验总说明、团队项目阶段表、评分 rubric、统一提交要求

### 章节级材料

以下内容围绕每章展开，但承担不同功能：

- `Technical Book Manuscript`：承载长期稳定、值得认真阅读的解释与案例，是主书稿正文
- `Student Study Guide`：承载本章学习路径、重点概念提醒、误区提示、阅读建议、与项目主线的连接
- `Instructor Guide`：承载本章教学重点、课堂组织建议、案例取舍、评价证据与实验映射
- `Appendices`：承载本章所需的命令速查、模板、配置片段、对照表等高操作性内容
- `Lab / Project Materials`：承载本章进入团队项目或实验体系时所需的任务说明与交付要求
- `Case Library`：承载本章可调用的真实仓库、历史事件、治理案例与项目证据

具体写作时应遵循以下规则：

- 所有材料层都应来源于同一份整体课程设计，而不是事后拼接
- `Technical Book Manuscript` 必须保持技术书的完整性，不承载课堂行政与评分说明
- `Technical Book Manuscript` 必须默认面向技术读者，而不是显式面向学生或教师
- `Student Study Guide` 和 `Instructor Guide` 必须基于主书稿使用，而不是重写一套新的正文
- 与平台界面强相关、变化频繁的操作细节，优先放入 `Appendices` 或 `Lab / Project Materials`
- 与课程组织方式强相关的说明，优先放入 `Course Syllabus / Course Guide` 或 `Instructor Guide`
- 只有那些对理解开源软件开发长期有效、且值得独立阅读的内容，才进入 `Technical Book Manuscript`
- 快速变化但教学上必须提及的内容，可在主书稿中点到为止，并将细节下沉到附录、实验材料或案例库

## 章节结构

本课程采用“稳定内核 + 演化实践”的结构，前半部分回答“开源软件开发是什么、为什么这样组织”，后半部分回答“如何实践开源开发、AI 如何参与这套机制”。初步教材章节规划如下：

- 第 1 章：开源的起源与发展——从自由软件运动到开源运动的历史脉络、关键人物与代表性事件
- 第 2 章：自由软件、开源软件与许可证——核心概念、主要许可证类型、版权与合规基础
- 第 3 章：开源社区与治理——社区结构、协作逻辑、治理模型与决策机制
- 第 4 章：开源开发的基本工程流程——版本管理、分支策略、Issue、Discussions、Pull Request、Code Review、CI、规则集（rulesets）与发布
- 第 5 章：阅读与理解开源项目——如何评估、阅读和理解一个真实开源项目的代码、文档与社区
- 第 6 章：参与开源项目——从第一次贡献到持续参与的完整路径，包括贡献规范、社区沟通、评审反馈与仓库治理要求
- 第 7 章：AI 辅助的开源开发实践——用 AI 理解项目、辅助编码、补文档、补测试，以及在开源流程下管理 AI 生成内容
- 第 8 章：组织与发布你的开源项目——如何让自己的项目能够被他人参与和持续维护，并具备基本的安全与供应链意识

以上为初步规划，各章主题和拆分粒度可在编写过程中根据实际需要调整。

## 章节体例约定

为保证全书风格统一、便于教学与自学，每章建议尽量采用相对一致的组织方式。章节体例可根据内容性质微调，但原则上应包括以下要素：

- 学习目标
- 核心术语
- 正文主线
- 代表案例
- 常见误区或讨论点
- 本章小结
- 延伸阅读
- 与附录、实验或案例库的对应关系

其中，偏文化、历史和治理的章节可以弱化技术操作细节，偏流程和实践的章节则应加强案例、流程说明和实践支撑。

## 写作原则

为降低教材过时风险，正文优先写原则、流程和方法，少写短期内容易变化的产品界面和操作细节。以 GitHub 作为主要示例平台，但尽量描述通用的开源协作流程；将更新频率高的工具清单、链接和产品比较放在附录、补充阅读或配套资料中。

除上述原则外，本教材还强调以下几点：

- 每章都应有明确的学习目标，使学生知道学完后“能够做什么”。
- 教材内容应能够支撑与其教学目标相匹配的实践活动，但具体实验、作业和课堂任务不必直接写入教材正文。
- 案例应尽量真实，但章节内容不能依赖某一个具体项目或某一家公司的产品细节才能成立。案例项目的选取应兼顾语言与领域的多样性、社区活跃度与可观察性、适合本科生理解的规模，以及许可证类型的代表性，并尽量反映全球开源生态的广度。
- 内容难度应循序渐进，从理解项目、阅读文档、完成小贡献开始，再逐步过渡到组织和维护项目。
- 教材不仅要讲“如何参与开源项目”，也要讲“如何让自己的项目能够被他人参与和持续维护”。
- 合规、安全和学术诚信应作为底线要求，在许可证、来源说明、AI 使用和工程流程中得到体现。
- 开源历史、关键人物和代表性项目不只是背景材料，而是帮助学生理解开源精神、社区形成和技术演化的重要入口。
- 开源历史、关键人物和代表性项目的写法应服务于思想、制度和技术演化的解释，不应写成与教材主线脱节的轶事堆砌。
- 关键术语应尽量保留英文对照，帮助学生逐步适应真实开源世界中的英文文档、README、Issue、Pull Request 和许可证原文。

## 考核对齐原则

课程的学习成果、教材内容、实验安排和考核方式应形成闭环，而不是彼此分离。设计课程时应确保：

- 每章至少对应一个明确的学习成果
- 每层实践至少对应一种可观察、可评价的能力证据
- 课程考核能够覆盖文化理解、概念掌握、流程操作和实际贡献四个层面
- AI 相关内容的评价不仅看“是否使用了工具”，还应看是否遵守了开源流程、质量要求和合规约束
- 课程项目或综合实践应尽量覆盖现代开源开发的关键交付物，如治理文件、Pull Request 与 Review 记录、CI 工作流、基本安全策略或发布证据

考核可以由阅读分析、课堂讨论、实验、项目、文档提交、代码贡献和课程答辩等多种形式组成，但都应能够回到课程学习成果本身。

## 更新策略

为保证课程长期可持续使用，课程材料应采用分层更新策略：

- 教材正文低频更新，以保持稳定性和可复用性
- 附录和实验材料每学年或每轮教学后复核一次
- 平台界面、工具链、AI 工作流等快速变化内容，优先通过附录或在线补充材料更新
- 当外部平台、许可证生态或 AI 工具环境发生明显变化时，应优先检查附录、案例库和实验材料，而不是轻易重写正文

## 术语约定

正文中的核心开源术语（如 Repository、Issue、Pull Request、Merge、Fork、License、Commit、Branch、Release 等）首次出现时给出中英对照，此后正文中统一使用英文原词。翻译不统一或容易引起歧义的术语，优先保留英文。对于已有广泛接受的中文译法且不会引起混淆的术语（如"版本管理""持续集成"），可直接使用中文，首次出现时附注英文。

为保证未来双语版本一致，还应遵循以下规则：

- 每个核心术语应尽量维护唯一、稳定的中英文对应关系，避免同一概念在不同章节出现多种中文译法
- 章节标题、节标题、材料类型名称和交付物名称，应尽量建立统一的中英文对照表
- 若某一中文术语难以稳定对应英文，优先保留英文原词，并在中文中作解释性表达
- 后续若编写英文版，应以统一术语表和统一章节结构为基础生成，而不是逐章自由翻译

判断某项内容是否应纳入教材时，可以使用以下标准：

> 这部分内容是否直接帮助本科生在真实开源项目中更好地理解、参与、贡献、维护或组织项目？

如果答案是“是”，则应优先纳入；如果只是一般软件开发知识、且与开源场景联系不强，则不应占据教材主体。
