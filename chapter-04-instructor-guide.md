# Chapter 4 Instructor Guide

## 对应书稿与使用方式

- 对应章节：[04-participating-in-and-organizing-open-source-projects.md](04-participating-in-and-organizing-open-source-projects.md)
- 建议用途：课程前半部分的工程流程基础单元
- 默认按一个 `100` 分钟教学单元设计；如需拆分，可分为“变更控制主链”与“CI / 安全 / 发布基线”两次 `50` 分钟教学

## 教学定位

本章要帮助学生理解：现代开源开发的核心不是“公开改代码”，而是“每一组修改都要经过隔离、说明、审查、验证和整合”。

## 教学目标

- 让学生理解流程的真正对象是变更控制，而不是平台按钮。
- 让学生理解 `Issue`、`Branch`、`Commit`、`PR`、`Review`、`Checks`、`Merge`、`Release` 构成一条连续链条。
- 让学生理解 CI 与自动化门禁的工程意义。
- 让学生理解依赖、密钥和发布完整性为什么属于主流程。

## 教学重点

- 第 1 节中的“主线可被信任”必须讲透。
- 图 4-1 与表 4-1 是本章的结构骨架，应反复回到“每个对象在回答什么问题”。
- 第 3 节不要讲成平台操作流程，要讲成“如何让修改成为公共判断对象”。
- 第 5 节必须打通“安全检查”和“发布边界”。

## 易错点与难点

- 学生容易把流程理解成平台规定而非工程抽象。
- 学生容易把 Commit 当作保存点，而不是历史单元。
- 学生容易把 CI 当成“自动运行测试”，忽略门禁与一致性作用。
- 学生容易把 Release 视为打标签动作，忽略其对外承诺含义。

## 推荐课堂组织

- `10` 分钟：从“为什么不能直接改主线”开场。
- `20` 分钟：讲第 1 节，建立变更控制视角。
- `25` 分钟：用第 2-3 节讲 `Issue -> Branch -> Commit -> PR -> Review -> Merge` 主链。
- `20` 分钟：讲测试、CI 与自动化门禁。
- `15` 分钟：讲依赖、密钥、发布与版本边界。
- `10` 分钟：让学生把这条流程映射到自己的团队项目。

## 推荐讨论问题

- 为什么开放协作反而更需要保护默认分支？
- 为什么 Pull Request 不是“上传代码”，而是“提交一个公共判断对象”？
- CI 为什么不是可有可无的附加工具？
- 为什么“合入主线”不等于“已经准备好发布”？
- 哪些安全问题必须在日常变更流程中前移处理？

## 案例使用建议

- 适合用一个真实 PR、一个 CI 检查页面、一个发布说明片段来讲对象。
- 如使用真实仓库，优先选择流程清晰、检查对象明确的项目。
- 不建议把教学重点放在具体平台界面点击步骤上。

## 与项目 / 实验主线映射

- 团队项目应在这一章之后形成最小工作流约定。
- 团队至少应明确工作项记录、分支边界、PR 审查和基本检查方式。
- 如课程安排允许，可把“提交一次最小 PR 并完成一次评审”作为阶段性任务。

## 可评价证据

- 团队项目最小工作流说明。
- 一个真实 PR 的流程分析。
- 团队项目的评审清单、检查清单或发布清单草稿。

## 基线层与扩展层建议

- 基线层：Issue、Branch、Commit、PR、Review、Checks、Merge、Release 的主链逻辑。
- 扩展层：更细的 CI 设计、依赖审查、安全扫描、SBOM、构件证明与更复杂的发布治理。

## 教学结构参考

- [The Missing Semester](https://missing.csail.mit.edu/): 借鉴其“讲义 + 工具演示 + 练习”的组织方式，本章适合边讲流程边做对象级分析。
- [Missing Semester: Version Control and Git](https://missing.csail.mit.edu/2026/version-control/): 借鉴其围绕 `git` 数据模型、分支和协作练习组织学习的做法，把工作流讲成可操作对象链。
- [Cornell CS5152: Open-Source Software Engineering](https://www.cs.cornell.edu/courses/cs5152/2019sp/): 借鉴其项目制、团队制、导师制背景，把流程讲成真实团队的协作基础而不是个人工具技巧。
- [CS50 Syllabus](https://cs50.harvard.edu/college/2026/spring/syllabus/): 借鉴其“lecture + section + problem set + synthesis form”的结构，本章适合把主讲、讨论和项目化任务明确分层。
