# 第 1 章 案例教学包

## 对应章节与用途

- 对应书稿：[01-overview.md](01-overview.md)
- 配套学习指南：[chapter-01-study-guide.md](chapter-01-study-guide.md)
- 配套教学指南：[chapter-01-instructor-guide.md](chapter-01-instructor-guide.md)
- 本案例包用于帮助学生把第 1 章读成一条连续历史主线，而不是一组著名人物与项目名单。

## 主案例与辅助案例

| 案例 | 本章主要说明什么 | 推荐观察对象 |
| --- | --- | --- |
| GNU | 把不满与理念转化为持续工程计划 | GNU 计划、自由软件定义、系统性建设语言 |
| Linux | 说明“可用系统”如何让开源从理念走向大规模工程实践 | Linux 内核、维护者体系、长期协作结构 |
| Mozilla | 说明开放代码、公共资源与浏览器竞争背景 | Netscape 开源、Mozilla 基金会、Firefox 历史角色 |
| Apache | 说明社区化治理与组织载体如何支撑长期公共性 | Apache 社区、基金会、项目群结构 |
| GitHub | 说明平台化如何改变开源协作的入口和可见性 | 仓库、PR、Issue、平台化协作语境 |

## 推荐课堂使用方式

- 用 GNU、Linux、Mozilla、Apache 四个案例分别回答“谁提出问题、谁让系统可用、谁推动公共传播、谁让组织稳定”。
- 把 GitHub 放在章尾使用，用来收束“平台化”而不是提前抢走前半章的历史重点。
- 课堂上可要求学生把每个案例压缩成一句话，强制他们从“项目介绍”转向“历史功能判断”。

## 与书稿贯穿对象的配合

- GNU 对应正文前半部分关于“问题如何被正式提出”的主线，用来帮助学生理解为什么开源不只是共享习惯，而是制度与工程议题的开端。
- Linux 对应正文中的 Linus 历史片段和“可用系统实践”讨论，用来说明为什么理念要变成可运行、可维护的工程对象。
- Mozilla 与 Apache 对应正文后半部分关于公共传播、组织化与长期协作的讨论，帮助学生区分“开放资源”“开放开发”“开放治理”不是同一件事。
- GitHub 对应章末关于平台化协作与现代开源可见性的收束，用来说明为什么它具有阶段转换意义。

## 课堂对象绑定建议

- 图 1-1 固定绑定 GNU -> Linux -> Mozilla / Apache -> GitHub 的阶段推进，用来帮助学生按“问题变化”而不是按“人物知名度”理解历史。
- Linus 的历史片段固定绑定 Linux，用来承接“为什么可运行系统会改变开源实践规模”。
- 课堂比较问题固定使用 GNU、Linux、Mozilla / Apache 这三组对象，不建议再额外扩案例。

## 推荐讨论问题

- 如果没有 GNU，开源问题会不会只是零散不满而不是系统计划。
- Linux 为什么不是简单重复 GNU，而是补上了另一类关键条件。
- Mozilla 和 Apache 为什么都重要，但承担的不是同一种历史角色。
- GitHub 的意义更接近技术平台、协作入口还是开源生态的放大器。

## 与团队项目主线的连接

- 团队项目在这一章不需要先证明“功能很多”，而需要先说明“为什么这个项目值得开放协作”。
- 教师可要求学生借用本案例包的语言，把自己的项目归到某一种“公共问题类型”上，而不是泛化为“我们也想做一个平台”。

## 使用边界

- 不要把本章讲成传记课或年份记忆课。
- 不要提前把后部章节的现代 AI 案例带进来，削弱前四章的稳定核心。
- 不要把 GitHub 讲成第 1 章主角，它在本章主要承担“平台化收束”作用。
