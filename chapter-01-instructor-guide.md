# Chapter 1 Instructor Guide

## 对应书稿与使用方式

- 对应章节：[01-overview.md](01-overview.md)
- 建议用途：课程前半部分的历史与定位引导单元
- 默认按一个 `100` 分钟教学单元设计；如需拆分，可分为“历史主线”与“现代开源再定义”两次 `50` 分钟教学

## 教学定位

本章不是开源人物故事课，也不是项目编年史。它的任务是帮助学生建立一张稳定的历史地图，使后续许可证、治理和工程流程不至于变成孤立工具知识。

## 本章教学目标

- 让学生理解软件从共享到封闭不是道德叙事，而是技术、商业与制度边界共同变化的结果。
- 让学生理解 GNU 把问题从不满转化为系统性工程计划。
- 让学生看清 Linux、Mozilla、Apache 的历史作用差异。
- 让学生建立“现代开源 = 技术 + 制度 + 组织 + 工程”的总框架。

## 教学重点

- 第 1 节中的“收费”与“控制权收紧”必须区分。
- GNU 的关键意义是把问题正式提出并组织成持续建设。
- 第 3 节中的三个案例要讲“各自补上什么”，而不是讲项目八卦。
- 第 4 节要收束到现代开源的四层结构。

## 易错点与难点

- 学生容易把历史读成零散事件，忽略主线问题。
- 学生容易把“自由软件”“开源”“公开代码”混成一个词。
- 学生容易对早期开源历史做浪漫化理解。
- 学生容易把 `Microsoft + GitHub` 看成商业新闻，而看不到其历史象征意义。

## 推荐课堂组织

- `10` 分钟：从 `Microsoft 收购 GitHub` 开场，追问“为什么这件事有历史意味”。
- `25` 分钟：按图 1-1 讲“共享实践 -> 封闭化压力 -> GNU”。
- `25` 分钟：比较 GNU、Linux、Mozilla、Apache 的不同角色。
- `20` 分钟：讲 1998 年后的术语转变、平台化、基金会化、产业化。
- `20` 分钟：回到“现代开源四层结构”，并连接课程后续章节。

## 推荐讨论问题

- 如果软件只是商品，为什么后来还会不断出现“开放协作”的制度需求？
- GNU 的历史意义更接近技术项目，还是更接近制度宣言？
- Linux、Mozilla、Apache 分别解决了什么前一阶段尚未解决的问题？
- 今天为什么不能再把开源理解为“代码可见”？

## 案例使用建议

- 以 GNU、Linux、Mozilla、Apache、GitHub 为主案例。
- 本章不建议引入后部章节使用的现代 AI 案例，以保持前半部分的稳定核心。
- 讲人物时应围绕其历史作用展开，不做传记式延伸。

## 与项目 / 实验主线映射

- 团队项目在这一章不要求进入工程实现深水区，重点是明确“为什么做成开源项目”。
- 可要求每个团队提交一段简短的项目公共性说明，说明问题对象、开放价值和潜在协作对象。
- 本章更适合作为项目主线的“定位与认知校准”阶段。

## 可评价证据

- 一页历史主线图，要求写出阶段问题而不是只写年份。
- 一个简短的项目角色比较说明，例如 GNU vs Linux 或 Mozilla vs Apache。
- 团队项目的开源定位说明。

## 基线层与扩展层建议

- 基线层：共享到封闭、GNU、Linux、Mozilla、Apache、1998 年术语转变、现代开源四层结构。
- 扩展层：基金会化、产业化与平台化的更深入讨论，可结合更复杂案例或阅读补充材料。

## 教学结构参考

- [Stanford CS101](https://web.stanford.edu/class/cs101/): 借鉴其 `Lecture Notes + Readings + Written Exercises / Code Exercises` 的分层结构，把历史叙述拆成“主讲 + 阅读 + 复述任务”。
- [Stanford CS101 Syllabus](https://web.stanford.edu/class/cs101/syllabus.html): 借鉴其课堂中使用短示例和现场练习的做法，本章可用时间线重构和案例对比代替长篇讲述。
- [CS50 Syllabus](https://cs50.harvard.edu/college/2026/spring/syllabus/): 借鉴其 `lecture + section + form` 的结构，把本章概念讲解与课后综合问题分开处理。
