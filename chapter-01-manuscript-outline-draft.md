# 第一章《开源的起源与发展》书稿提纲（草案）

> 当前状态：基于内部章节包与定向研究形成的书稿提纲草案。本文档用于进入正式章节写作前的结构收敛，不替代最终书稿正文。

## 开场段（草案）

2018 年 10 月 26 日，Microsoft 完成了对 GitHub 的收购。这个事件之所以具有象征性，不只是因为交易规模巨大，更因为它把开源历史中的一条深刻张力压缩在了同一幅画面里：一家曾长期站在商业软件一侧、甚至公开与 Linux 和自由软件阵营对立的公司，最终收购了全球最重要的开源协作平台之一。要理解这一幕为什么会发生，不能只从商业并购的角度解释，更要回到一个更早的问题：软件为什么会从共享走向封闭，又为什么重新走向开放协作？本章要讨论的，正是开源在几十年间如何从一种边缘的技术与伦理主张，逐步发展为现代软件世界的基础设施。

## 本章将回答的问题（Questions This Chapter Answers）

- 在“开源”这个词出现之前，软件世界经历了怎样的变化？
- GNU 与自由软件运动提出了什么关键问题？
- Linux、Apache、Mozilla 等项目为什么具有历史象征意义？
- “open source” 这一术语为什么在 1998 年成为转折点？
- 为什么今天必须把开源理解为一种生态系统，而不只是代码公开？

## 正文主线

本章建议采用 4 个一级小节。

## 1. 从共享到封闭：软件为什么会成为一个历史问题

### 本节目标

说明开源并不是凭空出现的新理念，而是对软件世界历史变化的回应。

### 本节核心论点

- 在计算机早期，软件往往依附于硬件，源代码共享在研究机构和厂商环境中较为常见
- 随着软件系统复杂度和商业价值上升，软件逐步成为独立商品
- 软件商品化并非错误，但它改变了软件的使用、传播和修改边界，进而引出了新的冲突

### 本节建议写入的内容

- 20 世纪中期到 70 年代的软件分发背景
- UNIX 授权变化作为“共享 -> 限制”转折的代表性案例
- 为什么“软件开始收费”不仅是价格问题，更是控制权问题

### 本节不展开的内容

- 不详细展开知识产权制度条文
- 不把 UNIX 技术史写成独立小史

## 2. GNU 与自由软件运动：问题被正式提了出来

### 本节目标

说明 GNU 项目和自由软件运动的历史角色，不把它们与后来的开源运动简单混同。

### 本节核心论点

- GNU 项目的意义不仅在于“做一个操作系统”，更在于重新提出软件自由的问题
- FSF 的建立使这种主张具有了持续性的组织形式
- 自由软件运动提供了后来开源运动的重要背景，但两者的表达重点并不完全相同

### 本节建议写入的内容

- 1983 年 GNU 项目公告与 1984 年实际启动
- 1985 年 GNU Manifesto 与 FSF 的组织化意义
- 为什么自由软件运动首先是在回应软件封闭化和控制权问题

### 本节写法提醒

- 这一节讲历史角色，不提前展开许可证比较
- 可以简要点出 GPL 的历史意义，但制度细节留到第 2 章

## 3. 从理念到生态：Linux、Mozilla 与 Apache 如何改变了开源

### 本节目标

说明开源并没有停留在口号层面，而是在不同阶段形成了可用系统、企业开放源码事件和稳定社区组织。

### 本节核心论点

- Linux 让自由软件系统真正走向大规模可用
- Mozilla / Netscape 事件让开放源码进入更广泛的公共视野
- Apache 则说明开源进一步发展出了成熟的社区协作和组织支撑

### 本节建议写入的内容

- Linux 与 GNU 组件结合带来的历史意义
- 1998 年 Netscape 开放源码与 Mozilla 项目的形成
- Apache 从补丁协作走向基金会化

### 本节结构建议

- 先写 Linux：强调“可用系统”
- 再写 Mozilla：强调“传播与主流化”
- 再写 Apache：强调“社区与制度化”

### 本节不应做的事

- 不堆砌大量技术产品名单
- 不把云计算、移动互联网、AI 平台全部塞进本节

## 4. 1998 到今天：为什么“开源”成了现代软件世界的基础设施

### 本节目标

把历史叙述收束到今天，让读者理解为什么后续章节必须继续讨论许可证、治理与工程流程。

### 本节核心论点

- 1998 年不仅是一个术语变化点，更是一个组织化、传播化和产业化的加速点
- OSI、基金会、代码托管平台和企业参与共同塑造了今天的开源生态
- 今天谈开源，已经不能只谈“源代码开放”，还必须谈规则、组织、协作和工程机制

### 本节建议写入的内容

- “open source” 术语和 OSI 的形成
- 企业参与、基金会化与平台化的大趋势
- 以 GitHub 和 Microsoft / GitHub 事件回扣章首问题

### 本节收束句的功能

这一节结束时，应自然把读者带向后续章节：

- 第 2 章：如果开源不是口号，它的制度基础是什么
- 第 3 章：如果开源依赖协作，它的社区与治理如何运作
- 第 4 章：如果开源需要规模化协作，它的工程流程如何组织

## 本章小结（预期方向）

本章的小结不应只是重复时间线，而应明确给出一个判断：开源之所以重要，不是因为它让代码变得“看得见”，而是因为它改变了软件被创造、传播、维护和共同拥有的方式。只有理解了这段历史，才能进一步理解许可证为何重要、社区为何需要治理、工程流程为何必须公开透明，以及为什么在 AI 时代仍然不能跳过这些基础。
