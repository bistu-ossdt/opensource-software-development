# 第 6 章 实验与项目材料

## 对应材料

- 对应书稿：[06-participating-in-open-source-projects.md](06-participating-in-open-source-projects.md)
- 配套学习指南：[chapter-06-study-guide.md](chapter-06-study-guide.md)
- 配套教学指南：[chapter-06-instructor-guide.md](chapter-06-instructor-guide.md)
- 对应实验阶段：实验 5 外部项目理解或首次贡献
- 团队项目阶段：外部贡献入口设计与第一次参与演练

## 本章任务目标

- 把“想参与外部项目”落实为一次可解释、可判断、可执行的第一次贡献路径。
- 让团队理解“开发自己的项目”和“进入别人的项目”在边界判断、沟通方式和证据组织上的差异。
- 让团队开始为自己的项目设计真正面向外部贡献者的第一次进入入口。

## 必做基础项

- 选择 `1` 个外部开源项目，识别 `1-2` 个可能适合作为第一次贡献的问题或对象。
- 围绕候选对象写出《第一次贡献目标分析》，说明为什么它适合或不适合现在进入。
- 完成一次真实的第一次贡献尝试，至少留下公开认领记录，以及提交或 PR 证据之一；若真实公共记录尚未走到完整链条，则用补充样稿补齐 PR 描述、验证说明和 review 回复；若条件不具备，则完成一套完整的“模拟贡献包”，包括认领说明、提交信息、PR 描述、验证说明和 review 回复样稿。
- 团队为自己的项目写出《第一次贡献路径说明》，建议放入 `docs/first-contribution-path.md`。
- 团队整理 `3` 类适合新贡献者进入的候选任务，建议放入 `docs/first-contribution-tasks.md`。

## 选做提升项

- 完成一次真实 PR 并经过至少一轮公开 review 往返。
- 对比两个外部项目的第一次贡献入口，说明哪一个更适合作为初次参与对象。
- 为团队项目补充 `good first issue` 或等价的新手入口标记，并说明筛选依据。
- 补充《第一次贡献被拒绝后的处理说明》，把失败场景也纳入方法框架。

## 本章交付物

- 一份《第一次贡献目标分析》，建议命名为 `first-contribution-target-analysis.md`。
- 一套真实贡献证据或模拟贡献包。
- 一份《第一次贡献路径说明》，建议仓库路径为 `docs/first-contribution-path.md`。
- 一份《第一次贡献候选任务清单》，建议仓库路径为 `docs/first-contribution-tasks.md`。

## 提交证据

- 外部项目仓库链接与候选 `Issue` / 文档对象链接。
- 《第一次贡献目标分析》链接。
- 真实参与记录，或模拟贡献包中的认领评论、commit message、PR 描述与 review 回复样稿；若真实公共记录尚未收到 review，可附预备 review 回复样稿。
- `docs/first-contribution-path.md` 与 `docs/first-contribution-tasks.md` 链接。

## 评分尺（实验 5 的第 6 章部分）

总分按 `100` 分计算，固定分为 `5` 个维度，每项 `20` 分。

| 评分维度 | 权重 | 不达标（`0-9`） | 达标（`10-15`） | 优秀（`16-20`） |
| --- | --- | --- | --- | --- |
| 候选对象判断 | `20` | 只凭标签、直觉或个人兴趣判断，没有核查时机、边界和当前上下文 | 能检查活跃度、范围、验证方式与进入前提，并说明为什么值得或不值得现在进入 | 能综合时间信号、关联对象、判断语气和版本窗口，清楚说明“为什么是这个对象、为什么是现在” |
| 贡献包组织 | `20` | 只提交零散片段，未形成“认领 -> 提交 -> PR -> 回复”的对象链 | 已形成基本完整的贡献链条，主要对象齐全，可判断其进入路径 | 整条链条前后连贯，公共记录意识清楚，真实记录与补充样稿之间衔接自然 |
| 贡献说明质量 | `20` | 只能说“改了什么”，说不清为什么改、如何验证、故意没改什么 | 能说明 What / Why / Verification，并基本控制修改范围 | 说明清楚、边界明确，验证具体可复现，还能主动说明 Not included 与风险边界 |
| review 协作能力 | `20` | 把 review 当作否定，或回复脱离原公共记录 | 能针对意见继续在同一条 PR 或等价记录中推进，并说明要重新确认什么 | 回复具体、克制、可执行，能把修改、验证与后续动作重新收束到同一条公共记录中 |
| 团队项目外部入口设计 | `20` | 仍站在组内成员视角，外部人不知道从哪里开始 | 已给出第一次贡献路径、候选任务和最小进入规则 | 路径说明、候选任务和优先级清楚，明显体现“外部贡献者第一次进入”的视角 |

## 真实贡献与模拟贡献的等价判定

- 是否被 merge、是否得到维护者即时回应，不作为本章直接加分项。
- 真实贡献如果尚未走到 review 阶段，不应因为“外部人还没回复”被机械扣分；教师应根据其公开记录、PR 说明和补充样稿判断其方法是否成立。
- 模拟贡献包只要明确基于真实公开对象、边界判断成立、贡献链条完整，也可以得到高分。
- 纯粹脱离真实上下文拼接的样稿，或与目标项目对象无法对应的“空心示例”，不应评为高分。

## 总体判分建议

- `90-100`：优秀。判断清楚、贡献链完整、说明具体，且能把方法迁移到团队项目。
- `75-89`：达标偏强。主要对象齐全，逻辑成立，只有少量边界或表达问题。
- `60-74`：基本达标。已形成完整进入路径，但判断深度、说明质量或团队迁移仍较弱。
- `60` 以下：不达标。仍停留在零散片段，或无法证明“为什么这个对象适合作为第一次贡献”。
- 若缺少《第一次贡献目标分析》、完整贡献包、或团队项目入口设计三项中的任一项，通常不判为“完整达标”。

## 完成度示意

- 最低通过示意：能说明目标为什么适合进入，提交完整的最小贡献包，并给出团队项目的第一次贡献路径与候选任务。
- 优秀完成示意：不仅对象齐全，而且能说明为什么这样组织、为什么不那样组织，以及外部贡献者进入自己项目时最先该看到什么。

## 常见扣分点

- 只凭 `good first issue` 或 `help wanted` 标签就直接认定“适合作为第一次贡献”。
- PR 描述只写“改了什么”，没有验证方法和边界说明。
- review 回复脱离原 PR 或只写“已修改”，没有明确回应具体意见。
- 团队项目的入口设计仍使用大量内部成员默认知识，外部人无法按文档真正进入。
- 贡献范围明显过大，却没有主动收紧边界或说明 Not included。

## 检查点

- 是否真正区分了“我想改什么”和“项目现在愿意接受什么”。
- 候选任务是否边界清楚、验证明确、范围可控。
- 模拟或真实贡献是否包含认领、说明、验证与 review 回复，而不只是代码修改。
- 收到反馈时，是否知道应继续在同一条公共记录中推进，而不是另起一套说明。
- 团队项目是否已经开始从“组内协作”转向“外部人如何进入”的视角。

## 常见问题

- 只凭 `good first issue` 标签就直接开工，没有再核查当前上下文。
- 把第一次贡献理解为“必须提交代码”，忽略文档、测试、问题分诊等更稳妥入口。
- 贡献包只写“改了什么”，没有说明为什么改、如何验证、故意没改什么。
- 把模拟贡献做成孤立片段，没有形成“认领 -> 提交 -> PR -> review 回复”的完整链条。
- 团队项目仍只考虑内部成员使用，不考虑外部贡献者第一次进入时需要看到什么。
