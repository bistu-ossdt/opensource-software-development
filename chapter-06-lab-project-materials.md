# 第 6 章 实验与项目材料

## 对应材料

- 对应书稿：[06-participating-in-open-source-projects.md](06-participating-in-open-source-projects.md)
- 配套学习指南：[chapter-06-study-guide.md](chapter-06-study-guide.md)
- 配套教学指南：[chapter-06-instructor-guide.md](chapter-06-instructor-guide.md)
- 对应实验阶段：实验 5 外部项目理解或首次贡献
- 团队项目阶段：外部贡献入口设计与第一次参与演练

## 本章任务目标

- 把“想参与外部项目”落实为一次可解释、可判断、可执行的第一次贡献路径。
- 让团队理解“开发自己的项目”和“进入别人的项目”在边界判断、沟通方式和证据组织上的差异。
- 让团队开始为自己的项目设计真正面向外部贡献者的第一次进入入口。

## 必做基础项

- 选择 `1` 个外部开源项目，识别 `1-2` 个可能适合作为第一次贡献的问题或对象。
- 围绕候选对象写出《第一次贡献目标分析》，说明为什么它适合或不适合现在进入。
- 完成一次真实的第一次贡献尝试，至少留下公开 `Issue`、认领记录、提交或 PR 证据之一；若条件不具备，则完成一套完整的“模拟贡献包”，包括认领说明、提交信息、PR 描述、验证说明和 review 回复样稿。
- 团队为自己的项目写出《第一次贡献路径说明》，建议放入 `docs/first-contribution-path.md`。
- 团队整理 `3` 类适合新贡献者进入的候选任务，建议放入 `docs/first-contribution-tasks.md`。

## 选做提升项

- 完成一次真实 PR 并经过至少一轮公开 review 往返。
- 对比两个外部项目的第一次贡献入口，说明哪一个更适合作为初次参与对象。
- 为团队项目补充 `good first issue` 或等价的新手入口标记，并说明筛选依据。
- 补充《第一次贡献被拒绝后的处理说明》，把失败场景也纳入方法框架。

## 本章交付物

- 一份《第一次贡献目标分析》，建议命名为 `first-contribution-target-analysis.md`。
- 一套真实贡献证据或模拟贡献包。
- 一份《第一次贡献路径说明》，建议仓库路径为 `docs/first-contribution-path.md`。
- 一份《第一次贡献候选任务清单》，建议仓库路径为 `docs/first-contribution-tasks.md`。

## 提交证据

- 外部项目仓库链接与候选 `Issue` / 文档对象链接。
- 《第一次贡献目标分析》链接。
- 真实参与记录，或模拟贡献包中的认领评论、commit message、PR 描述与 review 回复样稿。
- `docs/first-contribution-path.md` 与 `docs/first-contribution-tasks.md` 链接。

## 检查点

- 是否真正区分了“我想改什么”和“项目现在愿意接受什么”。
- 候选任务是否边界清楚、验证明确、范围可控。
- 模拟或真实贡献是否包含认领、说明、验证与 review 回复，而不只是代码修改。
- 收到反馈时，是否知道应继续在同一条公共记录中推进，而不是另起一套说明。
- 团队项目是否已经开始从“组内协作”转向“外部人如何进入”的视角。

## 常见问题

- 只凭 `good first issue` 标签就直接开工，没有再核查当前上下文。
- 把第一次贡献理解为“必须提交代码”，忽略文档、测试、问题分诊等更稳妥入口。
- 贡献包只写“改了什么”，没有说明为什么改、如何验证、故意没改什么。
- 把模拟贡献做成孤立片段，没有形成“认领 -> 提交 -> PR -> review 回复”的完整链条。
- 团队项目仍只考虑内部成员使用，不考虑外部贡献者第一次进入时需要看到什么。
