# 第三章《开源社区与治理》书稿提纲（草案）

> 当前状态：基于内部章节包与定向研究形成的书稿提纲草案。本文档用于进入正式章节写作前的结构收敛，不替代最终书稿正文。

## 开场段（草案）

很多人第一次接触开源时，会把它想象成一种近乎自发的协作：代码放出来了，感兴趣的人自然会加入，问题会被公开讨论，好的修改会被自动吸收，项目就会持续成长。但只要真正观察几个经典项目，就会发现事情远没有这么简单。Linux 的内核补丁不会直接无序地涌向主线，而是沿着维护者链条逐步流转；Apache 的项目也不是靠“谁都可以拍板”来运作，而是依赖 committers、PMC 和长期形成的社区规范。开源之所以能够成为公共协作系统，不是因为它没有组织，而是因为它把组织方式尽量公开、可追溯、可继承地写了出来。本章要讨论的，正是开源社区如何形成角色分工、治理规则和长期可持续的协作结构。

## 本章将回答的问题（Questions This Chapter Answers）

- 为什么开源项目不能只靠“代码公开”而自动形成稳定社区？
- 使用者、贡献者、维护者、核心团队、PMC、steering council 这些角色分别意味着什么？
- 为什么公开、异步、可归档沟通是开源治理的基础条件？
- `CONTRIBUTING.md`、`CODE_OF_CONDUCT.md`、Issue / PR 模板、`SECURITY.md` 为什么属于治理入口？
- 为什么社区健康、维护者负担和企业参与透明度必须进入治理讨论？

## 正文主线

本章建议采用 4 个一级小节。

## 1. 开源社区不是代码的附属物

### 本节目标

先打掉一个最常见的误解：开源不是“仓库公开之后自然形成社区”，而是围绕公共对象建立起来的协作结构。

### 本节核心论点

- “大教堂与集市”仍有解释价值，但不足以概括今天的治理现实
- 开源的关键不是“没人管理”，而是“管理方式尽量公开化”
- 社区存在的意义，不只是帮助写代码，更是建立公共规则、公共记录和共同预期

### 本节建议写入的内容

- 为什么开放协作需要面对陌生人合作、上下文传递和责任边界问题
- 为什么治理是规模化协作的前提
- 为什么第 1、2 章的历史与许可证讨论会自然走向社区与治理

### 本节写法提醒

- 不把“集市模型”写成今天的总解释框架
- 不把社区写成抽象价值口号

## 2. 角色、权力与责任如何被组织起来

### 本节目标

让读者理解开源项目中的角色不是身份标签，而是责任、权限和信任的分层安排。

### 本节核心论点

- 开源社区通常存在从使用者到贡献者、评审者、维护者、核心治理者的梯度
- 不同项目会采用不同治理结构，但都要解决“谁有权判断、谁承担责任”的问题
- 治理成熟的标志之一，是角色如何进入、如何升级、如何被约束被写得足够清楚

### 本节建议写入的内容

- Linux：维护者链条、补丁流转、Linus 的主线角色
- Apache：user / developer / committer / PMC member
- Python：core team、steering council、雇主约束

### 本节结构建议

- 先写 Linux：强调维护者体系
- 再写 Apache：强调社区化治理
- 再写 Python：强调正式治理与权力边界

## 3. 讨论、贡献与决策为什么要公开、异步、可归档

### 本节目标

说明沟通媒介和治理文件为什么属于同一类基础设施，而不是两组无关内容。

### 本节核心论点

- 开源治理依赖公共记录，而不是私下协调
- 异步沟通适应全球协作，也为后来者保留上下文
- 治理文件的意义，是把参与路径、行为边界和敏感事项处理方式显式化

### 本节建议写入的内容

- 邮件列表、Issue、PR 评论、Discussions 作为治理媒介的差异
- `CONTRIBUTING.md` 的功能：贡献入口和规则说明
- `CODE_OF_CONDUCT.md` 的功能：行为边界、报告和执行路径
- `SECURITY.md` 的功能：把普通公开讨论与敏感漏洞报告分开
- Issue / PR 模板的功能：降低沟通噪声，提升公共记录质量

### 本节不应做的事

- 不写平台操作手册
- 不把行为准则写成礼貌倡议

## 4. 从治理文件到社区健康：项目如何保持可持续

### 本节目标

把第三章从“规则存在”推进到“规则如何支撑长期运行”，让治理与可持续性直接挂钩。

### 本节核心论点

- 项目热度不等于社区健康
- 维护者负担、响应速度、贡献者留存和权力集中度都属于治理问题
- 企业参与不是例外，真正的问题是透明度和边界设计

### 本节建议写入的内容

- Open Source Guides 对社区指标的基本提醒
- CHAOSS 作为社区健康观察框架
- 维护者可持续性与继任问题
- 企业参与、供应商中立和利益约束的基本判断

### 本节收束功能

- 为第 4 章工程流程提供治理前提
- 为第 5 章项目阅读提供评估维度
- 为第 8 章项目创建提供治理基线

## 本章小结（预期方向）

本章的小结应明确指出：开源的公开性并不意味着无组织，恰恰相反，它要求项目把原本可能隐藏在私人关系中的角色、规则、讨论和责任边界尽量公开化。Linux、Apache、Python 这些经典项目之所以能够长期运行，并不是因为它们没有权威，而是因为权威、责任和公共记录之间形成了相对清晰的结构。理解了这一点，读者才会知道为什么一个项目需要贡献指南、行为准则、报告渠道和治理文件，也才会理解为什么下一章讨论工程流程时，Pull Request、Review、CI 和发布不只是技术动作，而是治理安排在工程层面的延伸。
