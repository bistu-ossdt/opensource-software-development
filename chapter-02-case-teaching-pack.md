# 第 2 章 案例教学包

## 对应章节与用途

- 对应书稿：[02-development-of-open-source-software.md](02-development-of-open-source-software.md)
- 配套学习指南：[chapter-02-study-guide.md](chapter-02-study-guide.md)
- 配套教学指南：[chapter-02-instructor-guide.md](chapter-02-instructor-guide.md)
- 本案例包用于帮助学生把许可证讨论落到真实项目边界，而不是只记许可证名称。

## 主案例与辅助案例

| 案例 | 本章主要说明什么 | 推荐观察对象 |
| --- | --- | --- |
| Linux kernel（GPL） | Copyleft 边界与开放延续逻辑 | 仓库许可证、项目定位、为何强调开放延续 |
| Apache HTTP Server（Apache-2.0） | 更清楚的声明与较低门槛复用如何并存 | 许可证文件、基金会语境、项目长期复用场景 |
| Jekyll（MIT） | 低门槛复用与简单制度边界 | 许可证文件、项目用途、复用语境 |
| 无许可证公开仓库（反例） | “公开可见”不等于“可公开协作” | 仓库公开状态、缺失 `LICENSE` 的后果 |

## 推荐课堂使用方式

- 让学生先找 `LICENSE`，再讨论项目目标和制度边界，避免只看仓库是否公开。
- 用 Linux kernel、Apache HTTP Server、Jekyll 做三角比较，让学生说出它们各自更保护什么边界。
- 反例应作为短对照使用，重点讲清“无许可证为什么不是小问题”，不要把课堂重心放到仓库吐槽上。

## 与书稿贯穿对象的配合

- Linux kernel（GPL）对应正文中关于 Copyleft 边界和开放延续的讨论，用来说明“继续开放”在什么场景下成为核心诉求。
- Apache HTTP Server（Apache-2.0）对应正文中关于声明边界、专利语境和长期复用的讨论，用来帮助学生理解“开放”并不只有一种制度表达。
- Jekyll（MIT）对应正文中关于低门槛复用的讨论，用来让学生看到宽松式许可证在工具类项目中的常见位置。
- “无许可证公开仓库”反例对应正文中“公开可见不等于标准意义开源”的关键判断，用来承接 Stallman 历史片段之后的制度落地部分。

## 课堂对象绑定建议

- 表 2-1 固定绑定 Linux kernel、Apache HTTP Server、Jekyll 三角比较，不建议再铺开过多许可证。
- Stallman 历史片段固定用于引出“为什么制度边界不能缺席”，然后再进入具体许可证案例。
- “无许可证反例”只承担校正作用，不应取代三角比较成为课堂主线。

## 推荐讨论问题

- 为什么两个都叫“开源项目”，许可证选择却可能完全不同。
- 低门槛复用和开放边界延续分别适合什么类型的项目。
- 一个仓库公开了，但没有 `LICENSE`，为什么仍不足以构成标准意义上的开源协作对象。
- 课程团队项目更接近哪一种边界需求，为什么。

## 与团队项目主线的连接

- 教师可要求团队先用本案例包的比较框架说明“我们的项目更像哪一类边界选择”，再决定许可证。
- 课堂上可让学生把真实案例中的许可证逻辑迁移到自己的项目对象清点上，尤其是代码之外的图片、数据和模板。

## 使用边界

- 不要把课堂推进到复杂兼容性迷宫或法律争议细节。
- 不要把许可证讨论变成“热门排行榜”。
- 不要只讲代码许可，忽略图片、数据和文档素材来源。
