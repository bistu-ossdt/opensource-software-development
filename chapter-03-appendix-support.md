# 第 3 章 附录支撑

## 使用说明

- 本附录服务于第 3 章的治理基线与贡献入口任务。
- 这些内容应被视为“最小可协作集合”，目标是让陌生人看懂项目，而不是一次写出成熟社区的全部治理文件。
- 若项目仍很小，可先完成最小集合，再逐步扩展。

## 最小治理文件清单

| 对象 | 建议位置 | 作用 | 当前阶段是否必需 |
| --- | --- | --- | --- |
| `README` | 仓库根目录 | 告诉读者这是什么项目 | 是 |
| `LICENSE` | 仓库根目录 | 给出制度边界 | 是 |
| `CONTRIBUTING.md` | 仓库根目录 | 告诉贡献者从哪里进入 | 是 |
| Issue 模板 | `.github/ISSUE_TEMPLATE/` | 统一问题描述入口 | 是 |
| PR 模板 | `.github/PULL_REQUEST_TEMPLATE.md` | 统一变更说明入口 | 是 |
| `CODE_OF_CONDUCT.md` | 仓库根目录 | 给出行为边界 | 选做，但推荐 |
| `SECURITY.md` | 仓库根目录 | 指出安全问题报告方式 | 选做，但推荐 |

## 外部贡献入口检查清单

- 仓库首页是否说明了项目用途和当前状态。
- 贡献者是否知道从哪里报告问题、提出建议或提交修改。
- 团队内部是否已经把责任分配成了可执行的角色，而不是临时商量。
- 贡献路径是否能被公开记录，而不是完全依赖私聊。

## 快速提醒

- 不必把小项目伪装成大社区，但必须让最小协作入口成立。
- 模板和文件名不是目的，真正的目标是降低陌生人进入项目的摩擦。
- 如果安全问题没有正式通道，至少要给出最小联系人或最小报告路径。

## 示例包使用顺序

- 教师课堂校准时，建议先展示“常见问题样例”，再展示“达标样例”和“高质量样例”。
- 学生自查时，建议先独立写出角色分工和贡献入口，再来对照样例。
- 样例重点是帮助判断“陌生人能不能进入”，而不是要求文件越多越好。

## 示例包 A：高质量样例

### 《角色分工与贡献入口草案》片段

```md
## 外部贡献者从哪里进入

- 首先应该阅读什么：README、CONTRIBUTING.md 和 docs/setup.md
- 应该通过 Issue、Discussion 还是其他路径进入：功能问题与修复建议通过 Issue；方向性讨论先走 Discussion
- 适合新手从哪一类任务开始：文档修正、测试补充、小型 UI 文案修正

## 最小协作规则

- 分支、提交、PR 的基本要求：每项修改必须关联一个 Issue；PR 描述至少说明 What / Why / Verification
- 安全问题应如何报告：先通过 SECURITY 联系邮箱私下报告，不在公开 Issue 中披露细节
```

### 教师点评

- 这个样例的关键不是“对象多”，而是进入路径、协作规则和安全入口前后一致。
- 陌生人第一次进入时，已经能知道先看什么、往哪里提、什么适合先做。

## 示例包 B：达标样例

### 《角色分工与贡献入口草案》片段

```md
## 团队角色

- 谁负责方向判断：项目维护者
- 谁负责实现推进：功能开发负责人
- 谁负责评审或质量检查：评审负责人

## 外部贡献者从哪里进入

- 首先应该阅读什么：README 和 CONTRIBUTING.md
- 适合新手从哪一类任务开始：文档和小修复
```

### 教师点评

- 这类样例已经具备最小治理骨架，但责任和规则还可以再具体。
- 如果能进一步写清“通过哪里进入”和“安全问题怎么说”，会更稳。

## 示例包 C：常见问题样例

### 《角色分工与贡献入口草案》片段

```md
## 团队角色

- 组长：负责所有事情
- 组员：配合完成任务

## 外部贡献者从哪里进入

- 有兴趣的话可以联系我们
```

### 教师点评

- 这类样例的问题不在文件长度，而在于外部人仍然无法进入。
- “组长负责所有事情”不是角色分工；“可以联系我们”也不是公开协作入口。
