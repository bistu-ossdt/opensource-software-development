# 第 8 章 教学指南

## 对应书稿与使用方式

- 对应章节：[08-organizing-and-releasing-your-open-source-project.md](08-organizing-and-releasing-your-open-source-project.md)
- 建议用途：课程结课阶段的维护者视角与项目发布单元
- 默认按一个 `100` 分钟教学单元设计；如需拆分，可分为“项目公共界面”与“版本 / 维护 / 最小安全基线”两次 `50` 分钟教学
- 当前状态：初稿；已与正文、实验材料、附录支撑和案例教学包接通，可组织一轮完整教学

## 教学定位

本章不是“把仓库打包一下”的收尾课，而是帮助学生完成从贡献者视角到维护者视角的切换：一个项目怎样才真正能被外部世界理解、使用、参与和继续维护。

## 本章教学目标

- 让学生理解“公开仓库”与“可公开参与的项目”之间的差距。
- 让学生理解项目入口对象为什么必须从维护者视角被设计出来。
- 让学生理解版本、Release、Changelog 属于对外承诺。
- 让学生理解维护计划、治理路径与最小安全基线为什么决定项目能否持续存在。
- 让学生能把这些概念转化成最小可交付对象，而不是停留在抽象原则。
- 让学生能对团队项目完成一次发布前审查与最小整备。

## 本章与前后章的教学边界

- 本章承接第 7 章，进一步从维护者角度吸收前面所有章节的结果。
- 本章不展开复杂基金会治理、复杂供应链框架细节，只强调小型项目可落地的最小基线。
- 本章重点是把项目组织成公共对象，而不是继续讨论参与别人的项目。
- 本章不讲“如何运营社区增长”，也不讲复杂商业化发布流程。
- 如课堂上出现“如何把规则正式写进项目、如何组织发布门禁”等问题，应记录并在第 8 章继续展开。

## 教学重点

- 第 1 节必须让学生理解“公开可见”不等于“可公开参与”。
- 第 2 节要把 README、LICENSE、贡献入口讲成同一层公共界面。
- 第 3 节要讲透版本与发布的对外承诺性质。
- 第 4-5 节要把维护计划、治理路径、安全与供应链基线收束成项目持续性判断。
- 表 8-1 与图 8-1 要组合使用：前者负责对象分类，后者负责进入顺序。
- README 正例 / 反例、`CHANGELOG` 条目、维护计划与 `SECURITY.md` 片段要讲成“维护者真实产出对象”，而不是展示模板。

## 易错点与难点

- 学生容易把本章理解成仓库包装或排版问题。
- 学生容易低估版本说明和发布记录的对外含义。
- 学生容易觉得小项目不需要治理路径和安全基线。
- 学生容易认为“先把代码做完，其他文件以后再补”。
- 学生容易把“范围之外的请求”处理成情绪问题，而不是治理问题。
- 学生容易只检查文件是否存在，而不检查这些文件是否真的降低了外部人的不确定性。

## 课堂上不应展开过深的内容

- 不把课堂变成复杂 CI / SBOM / 证明体系工具配置课。
- 不展开大型基金会或企业级合规体系细节。
- 不把本章讲成单纯的 README 写作课。
- 不把课堂重点放在“网站好不好看”或“徽章多不多”，而要始终回到公共性与责任边界。

## 课前准备建议

- 教师至少准备一个“公共界面较清楚”的项目和一个“代码公开但入口薄弱”的项目，用于第 1 节快速对比。
- 课前让学生带着团队项目当前仓库状态进入课堂，至少能打开 README、LICENSE、Issue 区和版本记录。
- 如课堂时间允许，可事先要求学生提交一份“当前仓库公共界面自查表”，课堂中直接修订。

## 推荐课堂组织

- `10` 分钟：从“为什么很多公开仓库仍然无法被别人真正使用和参与”开场。
- `20` 分钟：讲第 1-2 节，建立项目公共界面意识，并结合表 8-1、图 8-1 讲清对象与顺序。
- `20` 分钟：讲第 3 节，把版本、Release、Changelog 拉回对外承诺。
- `20` 分钟：讲第 4 节，讨论维护计划与治理路径。
- `20` 分钟：讲第 5 节，建立小型项目的最小安全与供应链基线。
- `10` 分钟：让团队对照本章完成一次发布前检查。

## 推荐课堂活动

- 活动 1：给出两个仓库首页截图或摘录，让学生判断哪一个更接近“可公开参与项目”，并说明依据来自哪些对象。
- 活动 2：展示 README 正例与反例，让学生指出“哪个问题已经被回答，哪个问题仍留给外部人猜”。
- 活动 3：给出一组提交历史和一个 `CHANGELOG` 条目，让学生比较“提交记录”和“写给外部人的变更说明”有什么不同。
- 活动 4：让学生为团队项目写出一个五行维护计划，并标出其中最容易缺失的一项。
- 活动 5：给出一个不在当前版本范围内的需求，让学生练习写一段维护者回应，要求既不粗暴拒绝，也不模糊承诺。

## 推荐讨论问题

- 一个仓库为什么可能“代码公开”却仍然“难以参与”？
- README、LICENSE、CONTRIBUTING 分别在帮助外部人解决什么问题？
- 版本号和 Release 为什么不只是给作者自己看的记录？
- 为什么安全与供应链意识不能等项目变大后再补？
- 为什么对范围外请求说“不”也是维护者责任的一部分？
- 如果只能优先补两个对象，应该先补什么，为什么？

## 案例使用建议

- 适合用一个公共界面清楚的小项目和一个公共界面薄弱的项目做对比。
- 可直接用学生团队项目作为案例，让他们以维护者视角查缺补漏。
- 案例重点应放在“外部人能否理解并进入”，而不是“功能是否很多”。
- 课堂主案例不必追求大型知名仓库，小型但入口清楚的项目更适合学生模仿。

## 与团队项目主线的阶段映射

- 本章是团队项目从课程内产物走向公开发布对象的关键收束阶段。
- 本章结束后，团队至少应完成一次最小公开发布准备。
- 本章对应的阶段产物是《项目发布与维护基线说明》。
- 更理想的状态是：团队至少完成一次可被外部阅读的 README 整理、一次版本命名、一次变更说明和一份最小维护计划。

## 可评价证据

- 一份发布前检查清单。
- 团队项目的 README / LICENSE / 贡献入口整备结果。
- 一份最小维护计划与版本说明。
- 一份最小 `CHANGELOG` 条目或 release note。
- 一份最小安全策略说明或 `SECURITY.md` 草稿。

## 最小达标证据 / 提升证据

- 最小达标证据：能提交《项目发布与维护基线说明》，并解释项目公共界面的基本组成。
- 提升证据：能将版本、维护计划、安全基线与后续治理路径组织成较完整的维护者方案。

## 评分关注点建议

- 首先看学生是否真的完成了“从贡献者到维护者”的视角切换，而不是只补了几个文件名。
- 其次看公共界面是否回答了外部人最先关心的问题，而不是追求文档长度。
- 再看版本、变更说明与维护计划是否彼此一致，避免出现“有版本号但没有边界解释”的空壳状态。
- 安全与供应链部分按“是否建立最小基线”判断，不按工具堆砌多少判断。

## 常见反馈语言建议

- “你已经把文件补上了，但还没有真正回答外部人最先关心的问题。”
- “这份 README 说明了功能，却还没有说明项目当前状态和参与入口。”
- “你写了版本号，但还没有把版本号变成清楚的对外承诺。”
- “维护计划里有愿景，但还缺少谁负责、多久看一次问题、哪些版本还支持这类近距离信息。”
- “这份说明像内部团队记录，还没有完全转成写给陌生人的公共对象。”

## 基线层与扩展层建议

- 基线层：项目入口、README、LICENSE、贡献入口、版本 / Release、最小维护计划、最小安全基线。
- 扩展层：更完整的治理路径、发布自动化、依赖治理、来源说明与更系统的供应链实践。

## 教学结构参考

以下参考用于借鉴教学结构与支架方法，不用于照搬内容或作业设计。

- [Open Source Guides: Starting an Open Source Project](https://opensource.guide/zh-cn/starting-a-project/): 借鉴其从创建者视角组织项目公共界面的结构。
- [Cornell CS5152](https://www.cs.cornell.edu/courses/cs5152/2019fa/): 借鉴其把课程项目直接组织成真实开源发布对象的做法。
- [The Missing Semester](https://missing-semester-cn.github.io/): 借鉴其强调“把隐含知识显式化”的方法，用于解释为什么项目入口与维护说明必须写给后来者看。
- [GitHub Docs: Setting up your project for healthy contributions](https://docs.github.com/en/communities/setting-up-your-project-for-healthy-contributions): 借鉴其把项目入口对象组织成一组仓库健康信号的方式。
- [Semantic Versioning](https://semver.org/): 借鉴其把版本号理解为公开边界承诺的表达方式。
- [Keep a Changelog](https://keepachangelog.com/): 借鉴其把变更记录写给人读而不是写给提交历史看的组织方式。
